## 1. Prompt 设计的核心认识

Prompt 不是“随便问一句”，而是给模型下达任务、约束、目标和输出格式的一整套指令。

做产品时，Prompt 最重要的不是文案写得多漂亮，而是这 4 件事是否清楚：
- 任务目标是否明确
- 输出合同是否明确
- 业务规则是否明确
- 错误场景是否有兜底

一句话总纲：

`Prompt 设计不是写一句“好提示词”，而是把任务、输出、规则、样本、评测和 badcase 串成一个迭代闭环。`

## 2. 四类 Prompt 的设计目标与方法

### 2.1 分类类 Prompt

#### 目标

把输入内容稳定映射到一个封闭标签集合中，解决“这是什么问题”的判断问题。

典型场景：
- 评价主题分类
- 意图识别
- 工单类型归类
- 异常原因归类

#### 设计方法

分类类 Prompt 的核心不是“解释得多好”，而是“分类口径稳不稳”。

设计时优先抓 5 件事：
- 先把标签列表定死，不给模型自由发明新标签
- 写清每个标签的定义和边界
- 为多问题输入写优先级规则
- 为容易混淆的边界样本补 few-shot
- 把输出收死成单标签或固定字段

适合的输出形式：
- 单标签
- 单标签 + 固定理由字段
- JSON 枚举字段

设计重点：
- 标签必须是封闭集合
- 冲突样本必须有优先级规则
- 输出必须标准化，不能让模型换说法

一句话理解：

`分类 Prompt 的重点是先收住标签，再写清规则，再补边界样本，最后固定输出。`

#### 作业标准答案示例

下面这版来自 Day 1 练习 1，可作为分类类 Prompt 的参考例子：

```text
你是网络货运平台的订单评价分类助手。

任务：
根据用户输入的订单评价，判断其最主要的问题主题，并从以下标签中选择一个返回：

正向体验、时效问题、服务态度问题、货损货差问题、信息同步问题、其他

标签定义：
- 正向体验：整体满意，主要在表扬服务、时效、沟通或整体体验，且没有明确负向投诉
- 时效问题：主要抱怨到达晚、送货慢、延误、等待时间过长
- 服务态度问题：主要抱怨司机或客服态度差、沟通差、不耐烦、不专业、不配合
- 货损货差问题：主要抱怨货物破损、丢失、少货、错货、数量不对
- 信息同步问题：主要抱怨订单状态更新不及时、回单上传慢、进度信息缺失、不准确、系统显示与实际不一致
- 其他：以上都不明显，或信息不足，或无法判断主要问题

判定规则：
1. 只能选择一个最主要标签
2. 如果一条评价同时涉及多个问题，只返回用户最核心抱怨对应的标签
3. 如果评价前半句是礼貌性肯定，后半句是明确投诉，优先按投诉内容分类
4. 如果评价既提到“实际送达晚”，又提到“系统状态更新慢”，优先看用户抱怨重点：
   - 重点在“货到得晚”时，返回“时效问题”
   - 重点在“状态/回单/进度信息不准或不同步”时，返回“信息同步问题”
5. 如果评价主要是在表扬，没有实质性投诉，返回“正向体验”
6. 如果无法明确判断，返回“其他”

示例1：
订单评价：司机很准时，沟通也顺畅，货物按时送到，整体体验不错。
输出：正向体验

示例2：
订单评价：明明约好下午2点送达，结果晚上8点才到，严重影响收货安排。
输出：时效问题

示例3：
订单评价：货早就到了，但系统里一直没更新签收状态，查进度很不方便。
输出：信息同步问题

示例4：
订单评价：整体还可以，就是回单上传太慢，想查进度的时候信息不完整。
输出：信息同步问题

输出要求：
1. 你的回答必须且只能是以下六个标签之一：
正向体验、时效问题、服务态度问题、货损货差问题、信息同步问题、其他
2. 不要解释原因
3. 不要输出标点、引号、序号、换行或额外文字
4. 不要输出相近但不在标签列表中的词
```

### 2.2 建议类 Prompt

#### 目标

基于输入内容给出下一步动作建议，解决“接下来怎么处理”的问题。

典型场景：
- 客服跟进建议
- 运营处理建议
- 异常处理动作建议
- 人工兜底建议

#### 设计方法

建议类 Prompt 比分类类 Prompt 多了一层要求：不仅要识别问题，还要给出可执行动作。

设计时要重点约束 4 个维度：
- 建议必须具体、可执行
- 建议必须优先面向“下一步动作”
- 信息不足时不能乱猜，要先核验或补充信息
- 高风险问题要优先给出回访、核验、升级等保守动作

适合的输出形式：
- 固定段落格式
- 固定字段结构
- JSON schema

设计重点：
- 不允许输出“加强、优化、持续关注”这类空泛建议
- 需要明确谁来执行动作
- 需要明确信息不足时的默认策略
- 需要明确高风险场景的优先处理路径

一句话理解：

`建议类 Prompt 的核心不是“说得像人”，而是“建议能不能直接执行”。`

#### 作业标准答案示例

下面这版来自 Day 1 练习 2，可作为建议类 Prompt 的参考例子：

```text
角色：
你是网络货运平台的订单评价分析助手。

任务：
根据用户输入的订单评价，识别情感倾向、主要问题类型，并给客服或运营输出具体、可执行的跟进建议。

情感倾向标签定义：
- 正面：整体满意，主要在表扬服务、时效、沟通或整体体验，且没有明确负向投诉
- 负面：用户有明确抱怨、投诉或明显不满
- 中性：评价信息不足、表达模糊，或没有明确正负向倾向

问题类型标签定义：
- 时效问题：主要抱怨到达晚、送货慢、延误、等待时间过长
- 服务态度问题：主要抱怨司机或客服态度差、沟通差、不耐烦、敷衍、不专业、不配合
- 货损货差问题：主要抱怨货物破损、丢失、少货、错货、数量不对
- 信息同步问题：主要抱怨订单状态更新不及时、回单上传慢、进度信息缺失、不准确、系统显示与实际不一致
- 其他：以上都不明显，或信息不足，或无法判断主要问题

一、输出闭集约束
1. 情感倾向必须且只能是以下三个标签之一：正面、负面、中性
2. 问题类型必须且只能是以下五个标签之一：时效问题、服务态度问题、货损货差问题、信息同步问题、其他

二、真实性与保守性约束
3. 不要编造不存在的事实
4. 如果信息不足，明确指出还需要补充哪些信息
5. 如果建议涉及原因判断、责任判断或异常说明，优先先写“核验/确认”，不要默认直接解释原因或直接处理
6. 如果评价信息不足，优先输出“建议先记录，必要时再补充信息”，不要默认直接进入回访或升级处理流程

三、建议质量约束
7. 跟进建议必须具体、可执行、不能空泛
8. 跟进建议优先写客服或运营下一步可立即执行的动作
9. 不要输出“加强、优化、提升、持续关注”这类没有直接动作的空泛表述

四、业务判定规则
10. 如果评价明显正面且没有实质性投诉，问题类型返回“其他”
11. 如果是正面评价，跟进建议统一写为“无需跟进，建议记录为正向评价样本。”，待补充信息写“无”
12. 如果一条评价同时涉及多个问题，只返回最主要、优先级最高的问题类型
13. 如果同时涉及货损货差和其他问题，优先返回“货损货差问题”
14. 如果同时涉及“实际送达晚”和“系统状态更新慢”，优先按用户抱怨重点判断：
   - 重点在“货到得晚”时，返回“时效问题”
   - 重点在“状态/回单/进度信息不准或不同步”时，返回“信息同步问题”
15. 如果问题涉及投诉、货损、强负面体验或需要核验事实，优先给出“建议回访/核验/升级处理”的动作

五、输出格式约束
16. 请按“订单评价1、订单评价2……”逐条输出，不要合并分析
17. 每条评价都必须使用以下固定格式：
订单评价X：
·情感倾向：
·问题类型：
·跟进建议：
·待补充信息：
18. 不要输出额外解释，不要补充与固定格式无关的内容
```

### 2.3 检索问答类 Prompt

#### 目标

让模型仅基于已检索到的资料回答问题，解决“当前资料里到底怎么规定”的问题。

典型场景：
- SOP 问答
- 规则条款问答
- FAQ 知识问答
- 按地区、版本过滤后的业务问答

#### 设计方法

检索问答类 Prompt 的第一职责，不是“多会组织语言”，而是“守住资料边界”。

设计时要明确 4 个原则：
- 只能基于给定资料回答，不能补充资料外规则
- 资料不足时要明确说“不足”
- 资料冲突时要明确说“冲突”
- 需要区分“资料中明确写了”与“模型自己推断”

适合的输出形式：
- 直接答案 + 依据说明
- 结论 + 引用材料
- 固定问答模板

设计重点：
- 不要让模型把常识脑补成企业规则
- 不要在证据不足时给确定结论
- 尽量要求回答引用材料标题、来源或片段标识
- 对最新规则、责任边界、时间限制等高风险信息要更保守

一句话理解：

`检索问答 Prompt 的重点不是让模型更会答，而是让它只在证据内作答。`

#### 作业标准答案示例

下面这版来自 Day 2 练习 3 的版本 A，可作为检索问答类 Prompt 的参考例子：

```text
角色：
你是网络货运平台知识助手。

任务：
请仅基于给定资料回答用户问题，不要使用资料外的信息，不要补充或假设企业规则。

回答要求：
1. 优先依据资料中的明确内容直接回答问题。
2. 如果资料不足以支持明确结论，必须明确回答：“当前资料不足，无法确认”，不要编造。
3. 如果检索到多份资料，且内容之间存在明显冲突，必须明确说明“资料内容存在冲突，无法确认一致结论”，并分别说明冲突来自哪些材料，提供对应材料链接。
4. 如果资料之间不存在冲突，但存在适用条件、区域范围、版本差异、时间差异或业务前提不同，回答时必须明确说明这些差异，避免混淆。
5. 不要编造规则、时间、责任方、处理结论或资料中未明确说明的限制条件。
6. 回答尽量简洁，先给结论，再说明依据。
7. 回答后必须说明答案主要依据的具体材料名称，并提供对应材料链接。

输出要求：
- 能直接回答时：
  先给出简洁答案；
  再补充“依据材料：xxx（链接）”。
- 资料不足时：
  直接输出“当前资料不足，无法确认”；
  如有相关材料，可补充“已参考材料：xxx（链接）”。
- 资料冲突时：
  直接输出“资料内容存在冲突，无法确认一致结论”；
  然后列出冲突材料及链接。
- 不要输出与问题无关的额外解释。

资料：
{{retrieved_context}}

问题：
{{user_query}}
```

### 2.4 Wiki 类 Prompt

#### 目标

基于已经整理好的 wiki、summary、知识页进行汇总、解释和回答，解决“通常怎么理解、怎么总结、怎么对齐知识”的问题。

典型场景：
- 知识手册问答
- 规则总览总结
- 概念解释
- 多文档差异归纳

#### 设计方法

Wiki 类 Prompt 和检索问答类 Prompt 的区别，不是“搜不搜”，而是知识来源不同：
- 检索问答更偏“现查原文”
- Wiki 类更偏“基于已整理知识页回答”

设计时重点关注：
- 先说明回答依据是 wiki / summary 层，而不是原始条款逐字查询
- 输出应更强调总结、归纳、对比、结构化表达
- 如果 wiki 资料不足或冲突，也必须保守输出
- 涉及精确条款、最新版本、区域差异时，应提示回查原始规则

适合的输出形式：
- 总结版答案
- 对比表
- 主题总览
- FAQ 统一口径页

设计重点：
- Wiki 类 Prompt 更强调知识压缩与复用
- 不能把 wiki 的总结误说成“唯一有效原文”
- 遇到精确责任、时效、版本时要提示回源

一句话理解：

`Wiki Prompt 的重点是让知识更好用，但不能假装自己等同于原始规则。`

#### 作业标准答案示例

下面这版来自 Day 2 练习 3 的版本 B，可作为 Wiki 类 Prompt 的参考例子：

```text
角色：
你是网络货运平台知识整理助手。

任务：
仅基于给定资料，对用户问题进行总结、对比和归纳。
禁止使用资料外的信息，禁止补充、猜测或假设企业规则、责任划分、处理结论、时间限制、例外条件。

回答原则：
1. 如果现有资料不足以支持回答，必须明确输出“当前资料不足”，不得编造信息。
2. 如果多份资料之间存在直接且无法同时成立的表述差异，必须明确输出“资料内容存在冲突”。
3. 若资料之间只是适用条件、区域范围、版本、时间、业务前提不同，不视为冲突；必须明确说明这些差异及其适用范围。
4. 不得根据常识、行业惯例或隐含逻辑补全资料中未明确写明的信息。
5. 回答尽量简洁，优先给出结论，再给出依据。
6. 所有结论都必须能在资料中找到对应依据。
7. 回答末尾必须列出主要参考材料名称及链接。

输出格式：

一、如果资料不足
当前资料不足

已检索资料：
- 材料名称1（链接）
- 材料名称2（链接）

二、如果资料冲突
资料内容存在冲突

冲突点：
- 冲突点1：材料A 与 材料B 的表述不一致
- 冲突点2：材料C 与 材料D 的表述不一致

已参考资料：
- 材料名称1（链接）
- 材料名称2（链接）

三、如果可以回答
结论摘要：
- 结论1（依据：材料A）
- 结论2（依据：材料B、材料C）

资料依据：
- 材料A：与问题相关的关键信息
- 材料B：与问题相关的关键信息

如存在适用差异，请单独说明：
- 差异点1：适用区域/版本/时间/前提

已参考资料：
- 材料名称1（链接）
- 材料名称2（链接）

要求：
- 不要输出与问题无关的解释
- 不要输出资料外推断
```

## 3. 四类 Prompt 的差别

| 类型 | 主要解决的问题 | 目标输出 | 最重要的设计重点 |
| --- | --- | --- | --- |
| 分类类 | 这是什么问题 | 单标签 / 枚举字段 | 标签闭集、边界清楚、优先级稳定 |
| 建议类 | 接下来怎么处理 | 动作建议 / 固定结构 | 建议具体可执行，不乱猜 |
| 检索问答类 | 当前资料怎么规定 | 基于证据的回答 | 守住资料边界，资料不足就承认不足 |
| Wiki 类 | 通常如何理解和总结 | 总览、摘要、对比、知识页式回答 | 知识压缩复用，必要时回查原文 |

## 4. Prompt 最佳设计流程

这是 Day 1 和 Day 2 共通的方法主线，也是最值得复用的 SOP。

### 第 1 步：先定义任务，不要一上来就写 Prompt

至少先回答这 4 个问题：
- 这个能力要解决什么问题
- 输入是什么
- 输出是什么
- 输出结果后面要接什么流程

如果任务没定义清楚，后面很容易出现 Prompt 越写越长，但输出依然不稳。

### 第 2 步：先定义输出合同

先把交付结果定死，再写 Prompt。

要明确：
- 输出是标签、文本还是 JSON
- 字段有哪些
- 哪些字段必填
- 哪些字段是枚举值
- 什么情况下返回兜底值

一句话：

`先定接口，再写 Prompt。`

### 第 3 步：先定义评测标准

至少要看：
- 准确率怎么看
- 输出格式是否稳定
- 会不会输出额外解释
- 枚举值会不会跑偏
- 哪些 badcase 最重要

没有评测标准，Prompt 优化很容易变成“凭感觉改文案”。

### 第 4 步：准备样本

建议至少准备 5 类样本：
- 常规样本
- 边界样本
- 高风险样本
- 反例样本
- 低信息样本

样本来源建议：
- 真实数据打底
- 人工补边界样本
- badcase 持续反哺

### 第 5 步：写基础版 Prompt

基础版通常只要 4 块：
- 角色
- 任务
- 标签或字段定义
- 输出要求

先跑起来，不要一开始就堆太多规则。

### 第 6 步：跑测试，先看错误模式

重点不是“对了几条”，而是“错在哪类问题”。

常见错误模式：
- 标签混淆
- 输出不规范
- 同义词乱飞
- 多问题场景摇摆
- 边界场景不稳
- 资料不足时乱答
- 资料冲突时强行给结论

### 第 7 步：按错误补规则

Prompt 优化的核心不是润色，而是堵错误。

常见补法：
- 标签混淆：补标签定义
- 多问题摇摆：补优先级规则
- 格式跑偏：强化输出约束
- 同义词乱飞：重复枚举值
- 高风险乱答：补边界和兜底规则
- 资料外脑补：补资料边界规则

### 第 8 步：只在必要时补 few-shot

few-shot 适合补：
- 最容易混淆的边界样本
- 输出格式容易漂的地方
- 规则不复杂但表达容易变形的任务

few-shot 不该乱加，更不该用来替代规则定义。

### 第 9 步：用 badcase 持续迭代

每次 badcase 都追问 4 件事：
- 理想输出是什么
- 模型错成了什么
- 为什么会错
- 应该补哪类规则

### 第 10 步：达到可用后冻结版本

当以下条件基本满足时，就不要继续做“文学优化”：
- 常规样本稳定
- 边界样本可控
- 输出格式稳定
- 高风险场景有清晰兜底

此时更应该做的是：
- 冻结版本
- 小范围上线
- 持续收集新 badcase

## 5. Prompt 规则怎么写：从 5 类规则去考虑

这是 Day 1 里最值得直接复用的一套方法，特别适合分类类和建议类 Prompt，也可以迁移到知识增强场景。

推荐按这个顺序去想：

`先收住输出 -> 防止乱编 -> 提高建议质量 -> 写清业务规则 -> 固定交付格式`

---

### 5.1 第一类：输出闭集约束

要回答的问题：
- 模型允许输出哪些标签、字段值、枚举值
- 哪些词绝对不能输出
- 输出是否必须从候选集合中选

适用目的：
- 防止模型自造标签
- 防止分类结果漂移
- 防止字段值失控

写法示例：
- `primary_issue 只能是：正向体验、时效问题、服务态度问题、货损货差问题、信息同步问题、其他。`
- `只能输出一个标签，不要输出解释。`
- `不要输出不在标签列表中的词。`

---

### 5.2 第二类：防止乱编约束

要回答的问题：
- 信息不足时怎么办
- 资料缺失时怎么办
- 不能推断哪些内容
- 什么内容必须先核验再说

适用目的：
- 防止模型脑补
- 防止把常识当规则
- 防止在低信息场景下强行给结论

写法示例：
- `如果输入信息不足，优先输出“当前信息不足”，不要自行补充原因。`
- `请仅基于给定资料回答，不要使用资料外的信息。`
- `如果资料中没有明确说明，不要推断责任方、时间限制或处理结论。`

---

### 5.3 第三类：建议质量约束

要回答的问题：
- 输出的建议是否具体
- 是否可执行
- 是否面向下一步动作
- 是否避免空泛表达

适用目的：
- 提高建议类 Prompt 的可用性
- 避免模型输出正确但没法落地的废话

写法示例：
- `跟进建议必须具体、可执行、不能空泛。`
- `优先输出客服或运营下一步可立即执行的动作。`
- `不要输出“加强、优化、提升、持续关注”这类没有直接动作的表述。`

---

### 5.4 第四类：业务判定规则

要回答的问题：
- 多问题场景怎么判主问题
- 冲突信息按什么优先级处理
- 正面、负面、强风险场景怎么分流
- 哪些边界场景有特殊规则

适用目的：
- 统一业务口径
- 提高边界样本稳定性
- 降低模型在冲突场景中的摇摆

写法示例：
- `如果同时提到多个问题，按负向影响最大的主要问题分类。`
- `如果前半句是礼貌性肯定，后半句是明确投诉，优先按投诉内容分类。`
- `如果问题涉及投诉、货损、强负面体验或需要核验事实，优先给出“回访/核验/升级处理”的动作。`

---

### 5.5 第五类：输出格式约束

要回答的问题：
- 最终应该长什么样
- 是否必须是 JSON
- 是否必须逐条输出
- 是否允许额外解释、Markdown、引号、序号

适用目的：
- 让结果可直接接产品流程
- 提高机器可解析性
- 降低前端、后端接入成本

写法示例：
- `只输出合法 JSON，不要输出 markdown。`
- `请按“订单评价1、订单评价2……”逐条输出，不要合并分析。`
- `不要输出额外解释，不要补充与固定格式无关的内容。`

---

## 6. 这 5 类规则怎么迁移到四类 Prompt

### 分类类

最重要的 5 类规则映射：
- 输出闭集：标签必须封闭
- 防止乱编：信息不足时输出兜底标签
- 建议质量：通常不重点使用
- 业务判定：边界与优先级最关键
- 输出格式：单标签或固定字段

### 建议类

最重要的 5 类规则映射：
- 输出闭集：情感、问题类型、动作类型可枚举
- 防止乱编：原因不明先核验
- 建议质量：最关键
- 业务判定：高风险问题优先升级
- 输出格式：固定段落或 schema

### 检索问答类

最重要的 5 类规则映射：
- 输出闭集：可用于限定结论类型或固定回答结构
- 防止乱编：最关键，必须守资料边界
- 建议质量：不是重点，但可要求回答清晰简洁
- 业务判定：资料冲突、不足、版本差异的处理规则
- 输出格式：结论 + 依据 + 来源

### Wiki 类

最重要的 5 类规则映射：
- 输出闭集：可限定输出为总览、差异、FAQ 等结构
- 防止乱编：不能把 wiki 外的信息补进来
- 建议质量：体现在总结是否清楚、有结构
- 业务判定：涉及精确条款时提示回源
- 输出格式：总览、对比表、要点式知识页

## 7. 四类 Prompt 的最小模板

### 7.1 分类类模板

```text
你是[业务场景]的分类助手。

任务：
根据用户输入的[文本类型]，判断其最主要的问题标签。

标签列表：
- [标签A]
- [标签B]
- [标签C]

判定规则：
1. [标签边界规则]
2. [多问题优先级规则]
3. [边界场景规则]

输出要求：
1. 只能输出一个标签
2. 不要输出解释
3. 不要输出标签列表之外的词
```

### 7.2 建议类模板

```text
你是[业务场景]的分析与跟进助手。

任务：
根据输入内容，识别主要问题，并给[客服/运营/审核人员]输出下一步可执行建议。

规则：
1. 建议必须具体、可执行
2. 优先输出下一步动作
3. 如果信息不足，先输出核验或补充信息建议
4. 如果涉及高风险问题，优先输出回访/核验/升级处理

输出格式：
[问题类型]：
[跟进建议]：
[待补充信息]：
```

### 7.3 检索问答类模板

```text
你是基于企业知识库回答问题的助手。

任务：
请仅基于给定资料回答用户问题，不要使用资料外的信息。

规则：
1. 如果资料不足，直接输出“当前资料不足，无法确认”
2. 如果资料冲突，直接输出“资料内容存在冲突，无法确认一致结论”
3. 不要补充资料中未明确说明的规则、责任、时间或限制条件
4. 回答时优先引用对应资料名称或来源

输出格式：
结论：
依据：
来源：
```

### 7.4 Wiki 类模板

```text
你是企业知识手册整理与回答助手。

任务：
请基于给定的 wiki / summary 资料，对问题给出清晰、结构化的总结回答。

规则：
1. 只基于给定资料回答
2. 如果资料不足，明确说明“当前资料不足”
3. 如果不同资料之间存在明显冲突，明确说明“资料内容存在冲突”
4. 如果问题涉及最新规则、精确条款、区域差异或责任边界，提醒回查原始规则

输出格式：
主题总结：
关键要点：
适用范围：
如需回查原文：
```

## 8. 最后总结

如果只记住这份文档里的 4 句话，最够用的是：

1. `先定任务和输出合同，再写 Prompt。`
2. `Prompt 优化不是润色文案，而是按错误补规则。`
3. `分类看稳定，建议看可执行，检索问答看证据边界，Wiki 看知识复用。`
4. `写规则时优先从 5 类去想：输出闭集、防止乱编、建议质量、业务判定、输出格式。`

这也是 Day 1 和 Day 2 关于 Prompt 设计最核心的共通方法。
