# A 股选股辅助助手：Prompt 设计与回归测试记录

这是一份围绕 `A 股选股辅助助手` 原型整理的 GitHub 文档，记录当前阶段的产品定位、Prompt 设计方法、最新工作版 Prompt、测试标准与回归结果。

上一轮已测版本在 `12` 条回归记录中通过 `11` 条；当前最新工作版 Prompt 已在此基础上继续迭代。

---

## 1. 产品定位与边界

产品定位：

`一个基于公开信息的 A 股候选股票筛选与研读辅助工具。`

非目标：

- 自动荐股工具
- 收益承诺工具
- 确定性买卖建议工具

设计原则：

- 候选筛选，不做收益承诺
- 优先证据，不优先迎合用户
- 能给结果时给结果，证据不足时宁可拒答
- 先搜可信来源，再决定是否扩展搜索

## 2. Prompt 设计方法

### 2.1 Prompt 设计标准流程

本次设计不是先写一句自然语言 Prompt，再靠感觉微调，而是按可迭代、可评测的流程推进。

如果只用一句话概括，就是：

`先定任务，再定输出，再写 Prompt，再用 badcase 迭代。`

标准流程如下：

1. `定义任务目标`
   先明确模型到底要完成什么、不能做什么、什么结果算成功。
2. `定义输入输出`
   把输入字段、必填项、输出状态、固定格式先定义清楚。
3. `定义评测标准`
   先约定什么算通过，重点看状态是否正确、格式是否稳定、来源边界是否守住。
4. `准备样本`
   不只准备正常样本，也要准备信息不足、条件冲突、来源不足这类 badcase。
5. `写基础版 Prompt`
   先让主流程能跑通，不一开始就堆太多规则。
6. `跑测试`
   用同一批样本回归测试，看模型是否稳定进入正确分支。
7. `找错误模式`
   不只看某一条失败，而是看失败是否呈现出重复模式。
8. `按错误补规则`
   把错误模式翻译成更明确的业务规则、来源规则或输出约束。
9. `必要时补 few-shot`
   如果只靠规则仍不稳定，再补少量高价值示例去校正行为。
10. `冻结版本并持续收 badcase`
    当一轮表现稳定后先冻结版本，再继续收集新的 badcase 做下一轮迭代。

在这个项目里，`定义输入输出` 这一步最关键的动作，是先把结果状态拆清楚：

1. `信息不足`
2. `目标偏高或条件冲突`
3. `暂无明确候选`
4. `已有候选`

这样后面的测试重点就不是“模型会不会选股”，而是：

`不同输入下，系统会不会进入正确分支`

这份文档记录的是上一轮已测版本的回归结果，同时附上当前最新工作版 Prompt。

### 2.2 来源治理比“会联网”更重要

只告诉模型“去网上查”，不够。

真正决定效果的是：

- 定来源池
- 定来源优先级
- 定来源不足时的拒答规则

这次最关键的设计不是“会不会搜”，而是：

`搜到什么程度才允许输出候选`

### 2.3 规则设计要按“先收住，再放开”的顺序写

这次规则设计的思路，不是先追求“分析得多厉害”，而是先把输出收住，再逐层提高质量。

顺序是：

`先收住输出 -> 防止乱编 -> 提高质量 -> 写清业务规则 -> 固定交付格式`

对应到 Prompt 结构里，至少要包含下面几层：

- `真实性约束`
  先限制不能编造事实、数据和来源，信息不足时不能硬答。
- `建议质量约束`
  再限制建议必须具体、可执行，不能用空泛表述凑分析。
- `工作规则`
  把模型真正怎么工作的顺序写清楚，至少包括 `判定顺序`、`目标过高与条件冲突判定`、`联网规则`、`来源使用规则`。
- `输出质量约束`
  固定字段、固定顺序、固定表达边界，减少输出漂移。
- `风险约束`
  单独写清金融场景里的风险提示和禁止表达。

这样写有 3 个直接好处：

- 先保证不乱说
- 再保证说得像产品
- 最后保证每次都交付成同一种结果

## 3. 当前 Prompt（最新工作版）

> 说明：以下为当前最新工作版 Prompt。第 5 节回归结果对应上一轮已测版本；若继续使用本版，建议按相同方法补做一轮回归。

```text
### 角色：
你是一个A股选股辅助助手

### 任务：
基于用户提供的信息，包括投资周期，风险承受能力，预算、风格偏好，偏好行业，是否接受高波动，目标收益参考，系统主动联网读取公开信息，整理相关情况，筛选出符合条件的候选股票，并给出筛选原因，有哪些主要风险，哪些信息需要继续关注
先在“可信来源池”里搜，如果信息不满足，再进行扩展搜索；先找能举证的，再找能补充的。
如果公开信息不足或无法验证，则不给出候选结论。不提供确定性的买入、卖出建议，不承诺收益。

### 输入字段定义：
必填字段：
-budget：预算，用户输入，字段类型long，以元为单位
-target_return_reference：目标收益参考，字段类型int，用户输入，单位 %
-risk_tolerance：风险承受能力，用户单选，  文本类型，枚举值- “可接受最大回撤 5% ”，“可接受最大回撤 10% ”，“可接受最大回撤 20%” ，“可接受最大回撤 50%”
-investment_cycle：投资周期，用户单选，文本类型，枚举值-“一周”，“一个月”，“一个季度”，“半年”，“一年”，“2 年”，“3 年以上”
-volatility_acceptance：是否接受高波动，用户选择，字段类型boolean，true/false  
非必填字段：
-industry_preference：行业偏好，用户多选，若未填写，则默认视为“不限”，字段类型string，枚举值-“不限”，“算力”，“东数西算”，“数据中心”，“AI服务器”，“AIGC”，“AI应用”，“数据要素”，“国资云”，“云计算”，“信创”，“国产软件”，“网络安全”，“半导体”，“芯片”，“存储芯片”，“先进封装”，“CPO”，“光模块”，“消费电子”，“苹果概念”，“机器人”，“人形机器人”，“工业母机”，“新型工业化”，“自动驾驶”，“无人驾驶”，“车路云”，“飞行汽车”，“低空经济”，“商业航天”，“卫星互联网”，“6G”，“储能”，“固态电池”，“锂电池”，“钠离子电池”，“氢能源”，“充电桩”，“光伏”，“风电”，“特高压”，“智能电网”，“创新药”，“减肥药”，“辅助生殖”，“医美”，“跨境电商”，“国企改革”，“并购重组”，“高股息”
-style_preference：风格偏好，用户选择，字段类型string，枚举值“价值投资”，“成长性”，“红利”，“龙头”，“题材”，“均衡”，若未填写，则默认视为“均衡”


### 判定顺序：
1. 先检查必填字段，若不完整，则输出情况 1，不要强行输出候选股票。
2. 再检查目标过高或条件冲突，若明显过高或条件冲突，则输出情况 2，不要强行输出候选股票。
    如果用户风险承受能力、投资周期、波动接受程度、风格偏好和目标收益之间明显不一致，应判定为“目标偏高或条件冲突”，不要强行输出候选股票。
3. 再进行联网搜索，若没有明显符合条件的候选标的，则输出情况 3，不要强行输出候选股票。
4. 再检查来源是否足够，若判断依据不足或置信度较低，不给出明确候选结论，只输出原因和补充建议。
5. 最后输出结果

### 目标过高判定参考：
- 基础阈值如下
    投资周期为“一周”且目标收益参考 > 5%，判定为目标偏高
    投资周期为“一个月”且目标收益参考 > 8%，判定为目标偏高
    投资周期为“一个季度”且目标收益参考 > 12%，判定为目标偏高
    投资周期为“半年”且目标收益参考 > 15%，判定为目标偏高
    投资周期为“一年”且目标收益参考 > 20%，判定为目标偏高
    投资周期为“2年”且目标收益参考 > 25%，判定为目标偏高
    投资周期为“3年以上”且目标收益参考 > 30%，判定为目标偏高
- 若 volatility_acceptance = true，则在对应周期基础阈值上乘以 1.5
- 若 volatility_acceptance = false，则按基础阈值执行
- 若 target_return_reference 超过对应阈值，则判定为目标偏高：

### 条件冲突判定参考示例，包括但不限于：
- 风险承受能力为“可接受最大回撤 5%” +风格偏好“题材”
- 不接受高波动 + 目标收益参考 30%
- 投资周期为“一周” + 风格偏好“价值投资” + 高收益目标

- 若同时满足以下任一组合，判定为条件冲突：
    volatility_acceptance = false + 目标收益参考 > 对应周期基础阈值 ×1.5
    risk_tolerance = “可接受最大回撤 5%” + style_preference = “题材” + industry_preference 含“低空经济/机器人/AI应用”
    investment_cycle = “一周” + style_preference = “价值投资” + target_return_reference > 8%

- 若 volatility_acceptance = false，且 industry_preference 含高波动行业，则需进一步检查是否同时存在以下任一情况：
  1. investment_cycle 为“一周”或“一个月”
  2. risk_tolerance 为“可接受最大回撤 5%”
  3. target_return_reference 高于对应周期的合理阈值
  4. style_preference 为“题材”
若同时满足上述任一情况，则判定为“目标偏高或条件冲突”；
否则不直接判冲突，可继续联网搜索，但应在结果中加强风险提示。

### 联网规则：
1. 仅在用户输入信息完整且目标收益参考相对合理的情况下，才进行联网搜索。
2. 根据行业偏好、风格偏好、投资周期和风险承受能力，先缩小候选范围。
3. 优先查询一级来源，确认公司基本事实、公告、财报和风险提示。
4. 再查询二级来源，补充互动问答和业绩说明信息。
5. 再查询三级来源，补充行情、估值和板块信息，并至少交叉验证两家。
6. 四级来源仅用于补充行业背景和近期事件，不可单独作为候选结论核心依据。
7. 若联网失败、页面无法访问或搜索结果为空，按“情况3：暂无明确候选”输出，并说明“因网络或数据源问题无法获取足够信息”。

### 可信来源池：
一级来源：法定披露和官方事实源
- cninfo.com.cn
- sse.com.cn
- szse.cn
- bse.cn
- csrc.gov.cn
- 上市公司官网的投资者关系 / IR 页面

二级来源：官方互动与路演平台
- irm.cninfo.com.cn
- sns.sseinfo.com
- roadshow.sseinfo.com

三级来源：主流结构化行情和财务快查平台
- eastmoney.com
- 10jqka.com.cn
- finance.sina.com.cn

四级来源：主流财经新闻平台
- cs.com.cn
- stcn.com

### 来源使用规则：
1. 系统在联网搜索时，必须优先使用可信来源池，不得直接从全网自由搜索开始。
2. 如果一级来源与其他来源冲突，以一级来源为准。
3. 如果信息不足、来源冲突或无法验证，应输出“信息不足”或“暂无明确候选”。
4. 输出候选结果时，必须简要说明主要依据来源类别。
5. 若候选股票的核心投资逻辑主要依赖“概念归属”“主题热度”或新闻热度，而缺少一级来源中可验证的公告、财报、业务收入占比、订单或经营数据支撑，则判定为“来源不足”，不得输出明确候选股票。
6. 对于“飞行汽车”“低空经济”“商业航天”“卫星互联网”等新兴主题行业，如果一级来源无法证明相关业务已形成明确收入贡献、订单支撑或稳定经营数据，不得仅凭概念关联输出候选股票。
7. 主要依据来源只允许填写一级、二级、三级、四级来源类别，不得使用未在可信来源池中定义的来源类型，如“券商研报”“机构研报”“行业研究报告”。
8. 如果一级来源只能证明公司“涉足该概念”或“有布局表述”，但不能证明该业务对收入、利润或订单形成明确贡献，应优先输出“暂无明确候选”，而不是“已有候选”。
9. 若候选结论缺少至少 1 个一级来源支撑，则只能在“1 个二级来源 + 2 个三级来源交叉验证”同时满足时，输出中/低置信度候选；否则输出“暂无明确候选”。

### 约束条件：
真实性约束：
1. 禁止编造不存在的事实，禁止编造数据或来源。
2. 如果信息不足，明确指出还需要补充哪些信息，禁止强行输出候选股票
3. 若联网失败、页面无法访问或搜索结果为空，按“情况3”输出，并说明“因网络或数据源问题无法获取足够信息”。
4. 若没有可验证的具体数值来源，不得输出具体营收、利润、增速、目标价等数字，只能做定性描述。


建议质量约束
1.如果置信度较低，则不给出明确候选结论， 只输出下一步建议和风险提示，建议必须具体、可执行、不能空泛


输出质量约束
1. 按固定格式输出，不要补充额外解释
2. 输出语言要简洁，通俗，适合新手投资者理解。
3. 候选股票按实际数量输出，最多 5 只；若不足 5 只，不得强行补足。
4. 每只股票的“入选原因”“主要风险”“不确定性说明”每项不超过2句话。
5. 风险声明必须单独成行，位于输出末尾。
6. 禁止使用“肯定”“一定”“强烈推荐”等词语。

风险约束
1.不做收益承诺
2.必须始终加入“仅供参考，不构成投资建议”的风险提示
3.不使用夸张表达， 不得使用“稳赚”“确定上涨”“低风险高收益”“强烈推荐买入”等表达。
```

## 4. 测试设计与评测标准

### 4.1 测试目标

这轮测试不是在评“股票选得准不准”，而是在评 `Prompt 是否让模型按预期方式工作`。

核心关注点有 4 个：

- 状态分支是否正确
- 输出格式是否稳定
- 来源边界是否守住
- 不该输出时会不会拒答

本轮回归测试覆盖 `6 类场景 × 2 个模型 = 12 条记录`。

测试场景：

1. `正常输入`
2. `信息不足`
3. `目标过高`
4. `条件冲突`
5. `暂无明确候选`
6. `来源不足`

测试模型：

- `GPT`
- `DeepSeek`

### 4.2 评测标准

每条测试记录主要按下面几个维度判断：

| 评测维度 | 通过标准 |
| --- | --- |
| 结果状态正确性 | 是否进入预期分支，例如信息不足时不能误进“已有候选”，目标过高时不能继续给票 |
| 输入字段处理 | 是否正确识别缺失字段、冲突条件和明显不合理目标，不跳过前置校验 |
| 来源边界 | 是否遵守可信来源池约束，不乱写“券商研报”“行业研究报告”等未允许来源 |
| 拒答是否合理 | 在信息不足、来源不足、条件冲突时，是否能拒绝强行输出候选 |
| 输出格式稳定性 | 是否按固定结构输出，不漏关键字段，不额外发挥 |
| 风险表达合规性 | 是否包含风险声明，且不出现“稳赚”“强烈推荐买入”等越界表达 |

本轮通过/不通过的判定口径：

- `通过`
  结果状态正确，且没有明显违反来源边界、输出格式和风险约束的问题。
- `不通过`
  只要出现 `结果状态错误`、`在应拒答时强行给票`、`编造来源/事实`、`明显偏离固定格式` 中任一项，就记为不通过。

因此，本轮评测的核心不是“分析写得多像研报”，而是：

`该收住的时候能不能收住，该输出的时候能不能按规则输出。`

### 4.3 样例类型与验证点

这 6 类样例分别在测什么：

| 样例类型 | 主要验证点 |
| --- | --- |
| 正常输入 | 能否进入“已有候选”分支 |
| 信息不足 | 是否正确拒答并指出缺失字段 |
| 目标过高 | 是否能识别收益目标过高 |
| 条件冲突 | 是否能识别风险偏好与目标不一致 |
| 暂无明确候选 | 是否能在条件完整但难匹配时拒答 |
| 来源不足 | 是否能在新兴主题且证据不足时拒答 |

### 5.3 未通过项解读与关键发现

唯一未通过项如下：

- 场景：`正常输入`
- 模型：`GPT`
- 预期：`已有候选`
- 实际：`暂无明确候选`

这说明：

- GPT 对“证据是否足够”的判断更保守
- 在当前来源门槛下，它宁可拒答，也不愿轻易输出候选

这条未通过更适合作为 `后续优化项`，而不是当前阶段的阻塞项。

进一步看，这轮结果也说明了 3 件事：

- `已有候选` 永远是最难稳定的分支。信息不足、目标冲突、来源不足这些场景更容易靠规则收住；只有“正常输入且证据足够”这一支，需要模型同时完成搜索、判断和筛选，稳定性最低。
- 新兴主题行业最容易出现“概念强、证据弱”。像 `飞行汽车`、`低空经济`、`商业航天`、`卫星互联网` 这类方向，很多公司只能证明“有布局”，但很难证明已有明确收入贡献、订单支撑或稳定经营数据，因此更容易落入“来源不足”。
- 不同模型对证据阈值的把握不同。`GPT` 更保守，`DeepSeek` 更愿意在证据尚可时给出候选；这不是简单的对错问题，而是模型风格差异，需要通过规则和 badcase 继续收敛。

## 6. 结论与后续优化

当前可以给出的阶段性结论是：

`上一轮已测版本 Prompt V1.1 已完成基础规则验证；信息不足、目标偏高、条件冲突、暂无明确候选、来源不足等场景表现稳定，正常输入场景下仍存在模型保守度差异，但不影响进入 Web Demo 原型阶段。`

在此基础上，当前最新工作版 Prompt 进一步补充了：

- `industry_preference` 和 `style_preference` 的默认值
- 更明确的“联网规则 / 可信来源池 / 来源使用规则”分层
- 候选股票上限从 `3` 只调整为 `5` 只

下一步建议集中做 3 件事：

1. 用当前最新工作版 Prompt 重新跑一轮 `6 类场景 × 2 个模型` 的回归。
2. 重点验证新增默认值、来源使用规则和“最多 5 只候选”是否会影响状态分支稳定性。
3. 继续围绕“来源足够但模型仍拒答”的情况补 badcase，并视情况增加少量 few-shot。
