## 4. 核心判断框架

判断一个任务更适合 `Workflow`、`Agent` 还是 `Tool Calling`，可以先问自己 4 个问题：

1. 这个任务的步骤是不是固定的
2. 这个任务需不需要真实系统数据
3. 这个任务会不会边查边变
4. 这个任务是否必须由系统真正执行动作

一个够用的判断口诀是：

- 如果 `步骤固定`，优先看 `Workflow`
- 如果 `需要真实数据或真实动作`，必须看 `Tool Calling`
- 如果 `路径不固定、需要动态判断`，优先看 `Agent`

很多真实业务场景里，答案不是三选一，而是：

`Workflow 负责骨架，Agent 负责动态判断，Tool Calling 负责真实执行。`

## 5. 核心概念

### 5.1 Workflow 是什么

你可以先把 `Workflow` 理解成：

`一条预先设计好的固定流程。`

它通常具备这些特点：

- 步骤相对固定
- 路径清晰
- 规则明确
- 便于控制
- 适合高频、重复、稳定的流程

典型场景：

- 固定审批流
- 标准客服工单流转
- 异常自动分流
- 风险规则打标

### 5.2 Agent 是什么

你可以先把 `Agent` 理解成：

`一个围绕目标、上下文和工具，动态决定下一步怎么做的执行体。`

和 Workflow 相比，Agent 更强调：

- 动态决策
- 多步推进
- 按情况选择工具
- 面对不确定问题时继续推进任务

一句话理解：

`Agent 不是更会聊天，而是更会围绕目标行动。`

### 5.3 Tool Calling 是什么

`Tool Calling` 可以粗暴理解成：

`模型先说清楚“我要查什么 / 做什么”，再由系统真正执行。`

比如模型不是直接说：

`这单已经签收。`

而是先发起：

- `get_order_status(order_id)`

系统查完后，把结果返回给模型，模型再输出最终回复。

所以 Tool Calling 真正解决的是：

- 查真实数据
- 调真实系统
- 触发真实动作
- 让 AI 从“会说”变成“能做”

### 5.4 三者最容易混淆的地方

最常见误解有两个：

1. 把 `Agent` 理解成“任何带大模型的产品”
2. 把 `Tool Calling` 理解成“模型自己真的在执行”

更准确的理解是：

- `Workflow` 解决固定流程问题
- `Agent` 解决动态决策问题
- `Tool Calling` 解决真实执行问题

## 6. 三者怎么组合

三者不是互斥关系，而是经常组合出现：

| 维度 | Workflow | Agent | Tool Calling |
| --- | --- | --- | --- |
| 核心作用 | 管固定流程 | 管动态决策 | 管真实执行 |
| 更适合什么任务 | 分类、分流、规则打标、标准流转 | 多步诊断、补信息、复杂判断、策略生成 | 查订单、查状态、查规则、写记录、触发动作 |
| 优点 | 稳定、可控、易落地 | 灵活、能处理不确定问题 | 可接真实系统、结果更可信 |
| 主要风险 | 过于僵化 | 跑飞、越权、成本高 | 工具边界不清、误触发动作 |
| 是否需要人工兜底 | 高风险时需要 | 更需要 | 动作类工具必须重点控制 |

一句话总结：

`Workflow 管流程，Agent 管决策，Tool Calling 管执行。`

## 7. 货运订单评价场景里的应用拆解

为了避免概念悬空，可以把货运订单评价场景拆成几个典型应用：

### 7.1 差评归因助手

- 目标：把评价归因为时效、服务、回单、货损、沟通体验等问题
- 更偏：`Workflow + 模型分类`
- 原因：标签相对固定，路径明确，重点是分类稳定

### 7.2 异常归因与处理建议助手

- 目标：从评价出发，结合订单上下文和规则输出下一步建议
- 更偏：`Agent + Tool Calling`
- 原因：需要边查边判断，也需要真实数据和规则支持

### 7.3 高风险评价预警

- 目标：识别可能升级为投诉、赔付、客户流失的问题
- 更偏：`Workflow + 风险规则`
- 原因：高风险识别通常更适合清晰阈值和明确规则

### 7.4 自动分流到处理队列

- 目标：把评价自动分给客服、运营、质控、承运商管理
- 更偏：`Workflow`
- 原因：这是标准化流转问题，不一定要做成 Agent

### 7.5 承运商服务质量洞察

- 目标：按承运商、司机、线路聚合评价，寻找重复问题
- 更偏：`分析型 AI 应用`
- 原因：重点在归纳、聚合和洞察，不一定要求实时动作

## 8. 最小工具清单 V1

工具设计不要一开始就追求“大而全”。

一个最小可用工具集，先回答 3 个问题：

- `查什么`
- `从哪里查`
- `查完之后给谁用`

### 8.1 最小工具清单示例

#### 工具 1：`get_order_basic_info(order_id)`

- 作用：查询订单基础信息，用来补全上下文
- 输入参数：`order_id`
- 返回建议包含：发货地、收货地、客户类型、承运商、司机、下单时间、签收状态
- 触发条件：需要确认订单上下文时
- 风险边界：没有订单号时不能盲查

#### 工具 2：`get_order_status_timeline(order_id)`

- 作用：查询订单当前状态和关键流转时间
- 输入参数：`order_id`
- 返回建议包含：当前状态、关键节点时间、是否超时、预计或承诺时效
- 触发条件：评价涉及晚到、未送达、卡点异常时
- 风险边界：不能让模型猜测真实状态

#### 工具 3：`get_receipt_status(order_id)`

- 作用：查询回单是否上传、上传时间和缺失原因
- 输入参数：`order_id`
- 返回建议包含：上传状态、上传时间、缺失原因
- 触发条件：评价涉及回单问题时
- 风险边界：没有真实结果前不能下结论

#### 工具 4：`search_exception_sop(issue_type, region, customer_type)`

- 作用：检索异常处理规则和 SOP
- 输入参数：`issue_type`、`region`、`customer_type`
- 返回建议包含：处理步骤、升级条件、限制条件、人工介入条件
- 触发条件：需要判断下一步建议是否符合规则时
- 风险边界：规则冲突时优先人工确认

#### 工具 5：`create_manual_review_ticket(order_id, issue_type, risk_level, summary)`

- 作用：创建人工复核或人工客服介入任务
- 输入参数：`order_id`、`issue_type`、`risk_level`、`summary`
- 返回建议包含：工单号、处理队列、创建状态、优先级
- 触发条件：高风险、信息冲突、赔付争议、责任判定类问题
- 风险边界：只允许升级，不允许模型自动给最终赔付结论

### 8.2 一个够用的分类方法

工具至少要分成两类：

- `查询类工具`
- `动作类工具`

一般来说：

- 查询类工具重点控制 `查询条件是否充分`
- 动作类工具重点控制 `权限边界和人工确认`

### 8.3 哪些动作必须人工兜底

以下动作不建议直接交给模型自动闭环：

- 赔付结论
- 责任归属判定
- 对外承诺
- 大额补偿
- 投诉升级结论
- 任何会直接改变客户权益的动作

## 9. Tool Calling Prompt 模板

Tool Calling Prompt 的重点不是“鼓励模型多调用工具”，而是定义：

- 什么时候应该查
- 什么时候不能乱查
- 查完之后怎么组织输出
- 哪些场景必须停下来或转人工

### 9.1 最小可用版本

```text
角色：
你是网络货运平台的订单异常处理助手。

任务：
你需要根据用户提供的问题，决定是否调用工具查询订单状态、回单状态或规则信息，再输出处理建议。

工具使用规则：
1. 涉及真实订单状态、回单状态、规则差异时，优先调用工具，不要凭空假设
2. 如果用户没有提供必要信息，如订单号，先说明缺少信息，不要盲目调用工具
3. 如果已有信息足够，优先用最少的工具完成任务，不要重复调用
4. 如果涉及赔付、责任判定、投诉升级、对外承诺，不要直接给出最终处理结论，优先建议人工核验
5. 工具结果返回后，再基于结果输出建议

输出要求：
1. 先说明主要问题
2. 再说明判断依据
3. 最后给出下一步建议
4. 不要编造工具结果
5. 不要越权执行高风险动作
```

### 9.2 产品经理写 Tool Calling Prompt 时最容易漏掉的点

- 没写清调用前置条件
- 没写“最少必要工具原则”
- 没区分查询权和执行权
- 没写高风险场景的停手条件
- 没要求输出时保留依据

### 9.3 一个简单的自检标准

写完 Prompt 后，至少问自己这 5 个问题：

1. 缺订单号时，模型会不会乱查
2. 信息已经够了时，模型会不会还继续查
3. 高风险动作时，模型会不会越权
4. 最终结果里，能不能看出依据是什么
5. 如果工具结果冲突，模型会不会停下来转人工

## 10. Agent 产品最常见的风险

### 10.1 乱用工具

表现：

- 用户只是问规则解释，模型却先去查订单
- 明明只需要查一个状态，却把多个系统全查了一遍
- 缺少必要信息时仍然发起查询

怎么防：

- 写清每个工具的触发条件
- 写清“信息不足先补信息，不要盲查”
- 加上最少必要工具原则
- 限制最大工具调用次数

### 10.2 越权动作

表现：

- 直接承诺赔付
- 直接判定责任归属
- 直接关闭投诉
- 直接对外给客户承诺

怎么防：

- 明确区分建议权和执行权
- 高风险动作必须人工确认
- 不把敏感执行权直接暴露给模型

### 10.3 多步失控

表现：

- 模型不断查询，却没有收敛
- 明明信息不足应该追问，却硬着头皮继续猜
- 已经能输出建议了，还在补查低价值信息

怎么防：

- 定义清楚任务完成条件
- 限制最大步数和最大工具调用次数
- 信息不足时优先停下来补信息
- 冲突信息或低置信度场景直接转人工

### 10.4 结果不可追溯

表现：

- 只给结论，不给依据
- 看不出命中了哪条规则
- 看不出用了哪些工具结果

怎么防：

- 记录每次工具调用和关键判断
- 输出时强制包含“问题判断 + 判断依据 + 下一步建议”
- 对关键建议展示命中的规则和使用的数据来源

一句话总结：

- `乱用工具`：边界不清
- `越权动作`：权限不清
- `多步失控`：不会停
- `结果不可追溯`：说不清

## 11. 三个练习

### 练习 1：任务判断

请判断下面 5 个任务，更偏向哪一种：`Workflow`、`Agent`、`Tool Calling`。

1. 按固定规则把订单评价分类
2. 查询订单当前运输状态
3. 收到“客户投诉回单未上传”后，自动检查订单、回单、时效，再输出建议
4. 批量把异常工单按优先级分流到不同队列
5. 根据用户一句模糊投诉，先追问缺失信息，再判断问题类型并建议下一步动作

参考答案：

- 更偏 `Workflow`：1、4
- 更偏 `Tool Calling`：2
- 更偏 `Agent`，且通常要配合工具：3、5

- 工具结果回传给模型之后，模型怎么继续输出
- 什么是 `tool_choice`
- 为什么“工具调用”本质上是一个多步流程，而不是一次性返回答案

如果你是产品经理，不需要先把所有 API 参数背下来，但要能理解这条最小链路：

`用户问题 -> 模型判断是否调工具 -> 系统执行工具 -> 工具结果回传 -> 模型组织最终输出`

---

## 15. 这一节最该带走的一句话

`固定流程交给 Workflow，真实数据交给工具，动态决策才交给 Agent。`
