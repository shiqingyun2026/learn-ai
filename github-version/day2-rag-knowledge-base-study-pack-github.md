## 2. Day 2 最重要的主线

Day 2 真正要建立的，不是“会不会调一个检索 API”，而是下面这条判断链：

1. 这个任务是不是依赖企业自己的知识
2. 这个知识是不是会频繁更新
3. 这个任务更像“临时查资料回答”，还是“提前整理知识后复用”
4. 这个场景需要的是 `长上下文`、`RAG`、`Wiki` 还是 `Agent`
5. 如果回答错了，错在 `Prompt`、`检索`、`Wiki`、`Agent` 还是 `原始资料`
6. 如何把 badcase 变成下一次更好的知识、规则和流程

一句话概括：

`Day 2 讲的是“知识怎么被组织、调用、约束和持续优化”，而不是只讲 RAG 这个技术名词。`

## 3. 核心概念

### 3.1 RAG 是什么

`RAG = Retrieval-Augmented Generation`

先记最小定义：

`先检索外部知识，再基于检索结果生成答案。`

更完整一点：

`RAG 的本质是把“先检索外部知识，再基于检索结果生成答案”接到大模型推理流程里，让回答不仅依赖模型参数里的旧知识，也依赖当前、指定、可控的资料。`

### 3.2 RAG 解决的问题

RAG 解决的不是“让模型突然更聪明”，而是：

- 模型不知道企业私有知识
- 模型不了解最新规则
- 模型容易给出泛泛而谈的回答
- 模型难以给出带依据、可追溯的答案

标准表述：

`RAG 解决的问题是：解决大模型不知道企业私有知识、不了解最新资料、容易泛泛而谈、难以给出可追溯依据的问题，让回答更贴近企业场景、更可执行、更有依据。`

### 3.3 RAG 的最小工作流

一个最小 RAG 流程通常有 4 步：

1. 用户提问
2. 系统先去知识库里检索相关资料
3. 把检索结果和用户问题一起交给模型
4. 模型基于检索结果生成答案

压缩成一句话：

`检索负责找资料，生成负责组织回答。`

### 3.4 检索和生成的分工

`检索` 主要负责：

- 找到相关资料
- 尽量找全
- 尽量找准

`生成` 主要负责：

- 理解用户问题
- 综合检索结果
- 组织回答结构
- 控制语气、格式、边界和表达

标准表述：

`检索和生成的分工是：检索负责根据问题找到相关资料，并尽量找全、找准；生成负责理解用户问题，结合检索结果进行筛选、组织、归纳和表达，输出最终答案。`

### 3.5 什么是 chunk

`chunk` 就是把原始文档切成适合检索的小片段。

为什么它重要：

- 切得太大，噪声多，重点不集中
- 切得太小，上下文不完整，语义容易断裂

标准表述：

`chunk 会影响知识被切分和检索的方式。切得太大，信息噪声多；切得太小，上下文不完整。chunk 设计不合理，会直接影响检索结果和最终回答质量。`

### 3.6 什么是 chunk overlap

`chunk overlap` 是相邻两个文本块之间重复保留的那一段内容。

它的价值是：

- 保留前后文连续性
- 降低规则被切断的风险
- 提高完整语义被召回的概率

一句话：

`chunk overlap 不是浪费重复，而是在用少量冗余换更稳的检索效果。`

### 3.7 什么是 embedding

`embedding` 可以先理解成：把一段文本变成可比较语义相似度的表示。

它的作用不是“存知识”，而是让系统可以做语义检索：

- 用户说法和文档说法不完全一样
- 但系统依然有机会找到相关资料

### 3.8 什么是 vector store

`vector store` 可以理解成：存放 chunk、embedding 和 metadata 的检索层。

它通常负责：

- 存文本块
- 存 embedding
- 支持语义检索
- 支持 metadata 过滤

### 3.9 什么是 metadata

`metadata` 是给知识片段打上的结构化标签。

例如：

- `业务线 = 网络货运`
- `文档类型 = SOP`
- `地区 = 华南`
- `更新时间 = 2026-04-01`
- `规则主题 = 回单`

它为什么重要：

- 帮助系统过滤
- 帮助系统排序
- 帮助系统溯源
- 支持按地区、版本、时间、业务线做精确约束

标准表述：

`metadata 能给知识片段补充来源、时间、主题、部门、文档类型等信息，帮助系统更准确地检索、过滤、排序和溯源，从而提升回答的准确性和可追溯性。`

### 3.10 RAG 的两个基础评测维度

先记住两个词：

- `召回（recall）`
- `精度（precision）`

粗暴理解：

- `召回`：该找出来的，有没有找出来
- `精度`：找出来的这些，相关不相关

RAG 最常见的两类问题就是：

1. `召回不够`
2. `精度不够`

## 4. 六种路线怎么选

### 4.1 纯 LLM

适合：

- 通用常识问答
- 开放式表达
- 创意生成

特点：

- 简单、快
- 不依赖外部知识
- 容易泛化或幻觉

### 4.2 Prompt 塞规则

适合：

- 规则少
- 任务固定
- 短期 Demo 或 MVP

特点：

- 成本低
- 起步快
- 规则一多就难维护

### 4.3 长上下文

适合：

- 一次性任务
- 材料范围明确
- 需要整体阅读和综合分析
- 暂时不值得建长期知识库

标准表述：

`什么任务更适合长上下文：更适合单次任务、上下文边界明确、需要一次性读完一批材料再完成输出的任务，尤其是资料量中等、无需建立长期可复用知识库的场景。`

### 4.4 RAG

适合：

- 回答依赖企业私有知识
- 资料会更新
- 需要依据和可追溯性
- 回答不能只靠通用常识

标准表述：

`什么任务更适合 RAG：适合那些需要依赖明确资料、需要引用依据、需要结合私有知识或最新信息来回答的任务，比如知识库问答、制度问答、产品资料问答、客服辅助、企业 wiki 问答等。`

### 4.5 LLM Wiki

适合：

- 高频知识长期沉淀
- 原始资料太散
- 需要稳定口径和复用
- 需要形成总览页、差异页、术语页、专题页

标准表述：

`LLM Wiki 和 RAG 的核心差别是：Wiki 是把知识先沉淀、整理、结构化后再给模型用；RAG 是在回答时临时从原始或半结构化资料中检索相关内容，再交给模型生成。`

### 4.6 Agent

适合：

- 不只是回答，还要执行
- 需要多步判断
- 需要调用工具和系统
- 需要把“发现问题 -> 修知识 -> 再验证”自动化

一句话：

`长上下文、RAG、Wiki 更像“使用知识”的方式；Agent 更像“推动知识持续更新和执行任务”的实现层。`

## 5. 知识库为什么不能只是“把文档都丢进去”

这是 Day 2 很关键的一个点。

原始文档不等于可用知识，原因通常有这些：

- 文档结构不统一
- 信息密度差异大
- 一份文档里夹杂很多无关信息
- 同一个问题的信息散落在多份文档
- 没有 metadata 就难以过滤和排序
- 没有结构化整理就难以沉淀稳定口径

标准表述：

`为什么知识库不能只是“把文档都丢进去”：因为原始文档不等于可用知识，若不经过整理和结构化，容易造成检索效果差、回答不稳定、口径不统一。知识库需要经过切分、结构化、标签设计和规则设计，才能更好支持检索和生成。`

## 6. RAG 最常见的 badcase

高频 badcase 主要有这几类：

1. `没检索到关键资料`
2. `检索到了不相关资料`
3. `资料不全或过时`
4. `多个资料互相冲突`
5. `chunk 切坏导致语义断裂`
6. `模型基于不充分资料脑补`

标准表述：

`RAG 最常见的 badcase 包括没检索到关键资料、检索到了不相关资料、资料不全或过时、多个资料互相冲突，以及模型基于不充分资料进行脑补，最终产生看似合理但不准确的回答。`

再压缩一句：

`RAG 的常见 badcase，要么是没找对资料，要么是拿到资料后没守住边界。`

### 6.1 RAG 最大的风险点

最值得直接背的一句是：

`RAG 最大的风险点是：检索错了，后面全错。检索质量不足，导致模型基于错误或不充分的上下文生成“看似有依据、其实不可靠”的回答。`

---

## 7. RAG Prompt 最重要的边界

RAG Prompt 最重要的，不是让模型“更会说”，而是让模型守住资料边界。

它至少要做到：

1. 只基于检索到的资料回答
2. 资料不足时明确说“资料不足”
3. 不要把通用常识脑补成企业规则
4. 最好能说明依据来自哪些材料
5. 如有冲突或适用差异，要明确指出

标准表述：

`RAG Prompt 最重要的边界是什么：最重要的边界是让模型基于检索到的资料回答，不能脱离资料随意发挥；当资料不足或没有依据时，要明确说不知道或信息不足，不能编造。`

## 8. LLM Wiki 与自我进化

### 8.1 为什么整理成 Wiki 后会更好用

因为 Wiki 是提前整理好的知识资产，模型不需要每次都重新检索和理解原始资料，而是可以直接基于更标准化的知识页回答。

它通常带来这些优势：

- 更容易理解
- 更容易复用
- 更容易做跨文档整合
- 更容易形成统一口径

标准表述：

`为什么整理成 Wiki 后会更好用：因为 Wiki 是提前整理好的知识资产，模型不必在每次回答时都重新检索和理解原始资料，而是可以直接在相对标准化的答案基础上生成回复，因此通常更稳定、更高效、口径更统一。`

一个直观类比：

`Wiki 更像答案库，RAG 更像开卷考试。`

### 8.2 Wiki layer 怎么设计

Wiki layer 不该按“文档存放方式”设计，而要按“业务问题和使用方式”设计。

设计时优先考虑：

- 按业务主题组织
- 按高频问题组织
- 按流程和规则组织
- 做总览页、差异页、术语页、专题页
- 保留来源、更新时间和适用范围

标准表述：

`Wiki layer 怎么设计：Wiki layer 要按照业务场景和用户问题方式来设计，把原始资料整理成结构化、可复用、易检索的知识单元，保证内容清晰、口径统一、方便模型直接理解和调用。`

### 8.3 什么叫知识系统的自我进化

这里的“自我进化”不是模型自己神奇地变聪明，而是系统通过 badcase 闭环持续优化知识层、规则层和工作流层。

标准表述：

`什么叫知识系统的自我进化：知识系统的自我进化，是指系统根据过往问答中的问题和 badcase，持续优化知识内容、Wiki、知识库和检索策略，让后续回答越来越准确、完整和稳定。`

### 8.4 一个最小可落地的自我进化闭环

最小闭环可以拆成 6 步：

1. `记录`
2. `评估`
3. `归因`
4. `写回`
5. `验证`
6. `上线`

更展开一点：

1. 记录用户问题、模型回答、检索结果和人工修正
2. 判断这次回答对不对，有没有越权发挥
3. 归因到 `Prompt / RAG / Wiki / Agent / 原始资料`
4. 把修正写回对应层
5. 把这个 badcase 放进评测集重新验证
6. 小范围上线，必要时保留人工兜底

标准表述：

`一个最小可落地的自我进化闭环怎么跑：先收集线上 badcase，再做问题归因，判断是知识内容、检索、Prompt 还是生成问题；然后把对应内容写回 Wiki、知识库或规则中，再用题库评测优化前后效果，持续迭代。`

### 8.5 从产品经理角度还要补哪些自我进化知识

至少要补这几类：

- 怎么衡量知识系统效果
- 怎么整理 badcase 题库
- 怎么分析问题归因
- 怎么制定优化策略
- 怎么做评测验证
- 怎么做版本管理和回滚
- 怎么处理知识新鲜度
- 怎么控制自动化更新的风险

标准表述：

`从产品经理角度还要补哪些自我进化知识：产品经理还需要补充知识系统评估与迭代的方法，包括如何衡量知识系统效果、如何整理 badcase 题库、如何分析问题原因、如何制定优化方案，以及如何通过评测验证优化是否有效。`

### 8.6 怎么判断 badcase 应该写回哪里

最常见的写回位置有 5 个：

1. `Prompt`
2. `RAG`
3. `Wiki`
4. `Agent`
5. `原始资料`

最好记的判断口诀：

- 有资料但没说好，写回 `Prompt`
- 没找到或没找准，写回 `RAG`
- 高频稳定知识未沉淀，写回 `Wiki`
- 需要多步流程和工具协同，写回 `Agent`
- 源资料缺失或错误，写回 `原始资料`

更完整的标准表述：

`怎么判断一次 badcase 应该写回 Prompt / RAG / Wiki / Agent / 原始资料：要先做问题归因。若是问题理解错误，优先看模型理解能力或 Prompt 是否明确；若是问题理解对了但没找到材料，优先看 RAG、原始资料或知识库是否缺失；若是找到了材料但没说好，优先看 Prompt、回答规则或生成组织方式是否需要优化。`

## 9. 三个练习

### 练习 1：判断一个任务更适合哪种路线

请判断下面这些任务更适合：

- `纯 LLM`
- `Prompt 塞规则`
- `长上下文`
- `RAG / file_search`
- `LLM Wiki`
- `Agent`

题目：

1. `什么是 Prompt？`
2. `平台回单上传超时后应该怎么处理？`
3. `帮我想一个物流品牌 slogan`
4. `华南区域 2026 年最新赔付规则是什么？`
5. `把这段订单评价归类为哪种问题`
6. `把最近 6 份赔付规则更新记录整理成差异摘要`
7. `维护一份网络货运异常处理知识手册`
8. `查订单状态 + 查规则 + 生成处理建议 + 发起流转`

参考答案：

- `纯 LLM`：1、3
- `Prompt 塞规则`：5
- `RAG / file_search`：2、4
- `长上下文`：6
- `LLM Wiki`：7
- `Agent`：8

### 练习 2：设计一个最小可用物流知识层

假设你要做一个 `网络货运平台知识助手`，请设计：

- 最小 `raw sources`
- 最小 `wiki layer`
- 最小 `metadata`
- 最小 `路由规则`

你至少应该想到：

- `raw sources`：FAQ、SOP、赔付规则、操作手册、区域政策、异常工单复盘
- `wiki layer`：回单专题页、签收异常总览、赔付规则总览、区域差异页、术语表
- `metadata`：`category`、`topic`、`region`、`updated_at`、`version`、`effective_from`、`source_id`
- `路由规则`：
  - 最新规则查询 -> `RAG`
  - 少量资料对比总结 -> `长上下文`
  - 稳定知识解释 -> `Wiki`

### 练习 3：写一个最小可用的知识增强 Prompt

写一个 Prompt，让模型满足下面 4 条：

1. 只基于资料回答
2. 资料不足时不能编造
3. 能指出答案依据
4. 多份资料冲突时要明确提示

一个最小可用版本：

```text
你是企业知识助手。

请仅基于给定资料回答用户问题，不要使用资料外的信息。

如果资料不足以支持明确结论，请直接回答“当前资料不足，无法确认”。

如果资料之间存在明显冲突，请明确说明“资料内容存在冲突，无法确认一致结论”。

回答时先给结论，再说明依据材料。

资料：
{{retrieved_context}}

问题：
{{user_query}}
```

## 10. 做题标准答案速记

下面这些句子，可以直接作为 Day 2 高频题目的标准回答。

### 10.1 RAG 的本质

`RAG 的本质是：把“先检索外部知识，再基于检索结果生成答案”接到大模型推理流程里，让回答不仅依赖模型参数里的旧知识，也依赖当前、指定、可控的资料。`

### 10.2 RAG 解决的问题

`RAG 解决的问题是：解决大模型不知道企业私有知识、不了解最新资料、容易泛泛而谈、难以给出可追溯依据的问题，让回答更贴近企业场景、更可执行、更有依据。`

### 10.3 检索和生成的分工

`检索和生成的分工是：检索负责根据问题找到相关资料，并尽量找全、找准；生成负责理解用户问题，结合检索结果进行筛选、组织、归纳和表达，输出最终答案。`

### 10.4 什么任务更适合 RAG

`RAG 适合那些需要依赖明确资料、需要引用依据、需要结合私有知识或最新信息来回答的任务，比如知识库问答、制度问答、产品资料问答、客服辅助、企业 wiki 问答等。`

### 10.5 RAG 最大的风险点

`RAG 最大的风险点是：检索错了，后面全错。检索质量不足，导致模型基于错误或不充分的上下文生成“看似有依据、其实不可靠”的回答。`

### 10.6 什么任务更适合长上下文

`长上下文更适合单次任务、上下文边界明确、需要一次性读完一批材料再完成输出的任务，尤其是资料量中等、无需建立长期可复用知识库的场景。`

### 10.7 LLM Wiki 和 RAG 的核心差别

`LLM Wiki 和 RAG 的核心差别是：Wiki 是把知识先沉淀、整理、结构化后再给模型用；RAG 是在回答时临时从原始或半结构化资料中检索相关内容，再交给模型生成。`

### 10.8 为什么整理成 Wiki 后会更好用

`因为 Wiki 是提前整理好的知识资产，模型不必在每次回答时都重新检索和理解原始资料，而是可以直接在相对标准化的答案基础上生成回复，因此通常更稳定、更高效、口径更统一。`

### 10.9 什么叫知识系统的自我进化

`知识系统的自我进化，是指系统根据过往问答中的问题和 badcase，持续优化知识内容、Wiki、知识库和检索策略，让后续回答越来越准确、完整和稳定。`

### 10.10 怎么判断一次 badcase 应该写回哪里

`关键看归因：有资料但没说好，写回 Prompt；没找到或没找准，写回 RAG；高频稳定知识未沉淀，写回 Wiki；需要多步流程和工具协同，写回 Agent；源资料缺失或错误，写回原始资料。`
