## 3. 核心概念

### 3.1 RAG 是什么

`RAG = Retrieval-Augmented Generation`

你可以先粗暴地记成：

`先查资料，再让模型基于查到的资料回答。`

它解决的核心问题不是：

`让模型变得更聪明`

而是：

`让模型回答时更贴近你自己的知识、规则和最新资料。`

### 3.2 RAG 为什么重要

如果 `没有 RAG`，模型只能依赖它训练时学到的通用知识：
- 可能不懂你们平台的规则
- 可能不知道你们内部 SOP
- 可能回答泛化
- 可能产生幻觉

如果 `接了 RAG`，模型更像是：

`回答前先翻了你们公司的制度手册、FAQ、操作规范、赔付规则，再基于这些资料作答。`

### 3.3 RAG 的最小工作流

一个最小 RAG 系统，通常至少有这 4 步：

1. 用户提问
2. 系统先到知识库里检索相关资料
3. 把检索结果和用户问题一起交给模型
4. 模型基于检索结果生成回答

可以压缩成一句：

`检索负责找资料，生成负责组织回答。`

### 3.4 检索和生成各自负责什么

`检索` 负责：
- 找到相关资料
- 尽量别漏
- 尽量别拿错

`生成` 负责：
- 理解问题
- 综合检索结果
- 组织回答
- 控制语气、格式、引用方式

所以 RAG 出问题，不能一股脑怪模型。

你要先判断：
- 是 `检索没找对`
- 还是 `生成没说对`

### 3.5 什么是 chunk

`chunk` 就是把原始文档切成适合检索的小片段。`

为什么不能直接整篇文档一起检索？

因为：
- 太长，检索粒度太粗
- 很难精准定位相关内容
- 容易把无关信息一起带进去

所以 RAG 的常见做法是：

`先切块，再建索引，再按块检索。`

### 3.5.1 什么是 chunk overlap

`chunk overlap` 就是：相邻两个文本块之间，重复保留的那一段内容。`

你可以先把它记成：

`chunk size` 决定每块有多大，`chunk overlap` 决定相邻两块重复多少。

它为什么重要？

因为如果完全不重叠，关键规则刚好被切在边界上时，前后文就容易断掉。

比如原文是：

```text
签收后 2 小时内未上传回单，视为上传超时。
如果图片模糊，需要补拍后重新上传。
超过 24 小时仍未补传，转异常工单。
```

如果有 overlap，系统可能会这样切：

- chunk 1：`签收后 2 小时内未上传回单，视为上传超时。如果图片模糊，需要补拍后重新上传。`
- chunk 2：`如果图片模糊，需要补拍后重新上传。超过 24 小时仍未补传，转异常工单。`

这里重复出现的：

`如果图片模糊，需要补拍后重新上传。`

就是 `chunk overlap`。

它的价值是：

- 尽量保留上下文连续性
- 降低一句规则刚好被切断的风险
- 提高检索时把完整意思一起召回的概率

一句话记住：

`chunk overlap` 不是浪费重复，而是在用少量冗余换更稳的检索效果。

### 3.6 什么是 embedding

`embedding` 可以先理解成：把一段文本变成一组可计算相似度的向量。`

它的作用是：

`让系统不仅能按关键词找，还能按语义相关性找。`

比如用户问：

`回单一直没上传怎么办？`

知识库里即使没写“上传”这个词，而写的是：

`签收凭证回传延迟`

语义检索仍然有机会把它找出来。

### 3.7 什么是 vector store

`vector store` 可以理解成：存放可检索知识片段的容器。`

它通常负责：
- 存文本块
- 存 embedding
- 支持语义检索
- 支持 metadata 过滤

### 3.8 什么是 metadata

`metadata` 是给知识片段打上的结构化标签。`

比如一条物流规则文档，可以带：
- `业务线 = 网络货运`
- `文档类型 = SOP`
- `地区 = 华南`
- `更新时间 = 2026-04-01`
- `规则主题 = 回单`

它的价值是：

`让检索不只是“像不像”，还可以“先过滤再查找”。`

### 3.9 RAG 里最常见的两个评测维度

你可以先记这两个词：

- `召回（recall）`
- `精度（precision）`

粗暴理解：

`召回`：该找出来的，有没有找出来
`精度`：找出来的这些，相关不相关

RAG 最常见的两类错误就是：

1. `召回不够`
相关资料没找到

2. `精度不够`
找出来一堆看着像，但其实不够相关

### 3.10 什么任务适合上 RAG

适合 RAG 的任务，通常有这几个特征：

- 回答必须基于企业自己的知识
- 资料经常更新
- 回答不能只靠通用常识
- 需要引用规则、文档、FAQ、SOP
- 需要降低幻觉

例如：
- 物流平台异常处理 SOP 问答
- 赔付规则问答
- 回单规则问答
- 平台操作说明问答

### 3.11 什么任务不一定要上 RAG

这些情况不一定需要 RAG：

- 通用常识问答
- 创意生成
- 任务规则很少，直接写进 Prompt 就够
- 没有稳定知识源可接
- 回答不依赖企业资料

一句话记住：

`不是所有 AI 问答都要上 RAG，只有当“知识属于你自己”时，RAG 才真正有价值。`

### 3.12 2026 补充：RAG 不是唯一知识增强路线

截至 `2026-04-08`，行业里有一波很热的讨论叫 `LLM Wiki`。

你可以先把它理解成：

`不是每次提问都临时从原始文档里捞片段，而是让 LLM 持续把原始资料整理、汇总、交叉引用，沉淀成一个可维护的 wiki，再围绕 wiki 回答问题。`

它之所以会火，主要因为：

- 长上下文模型更强了
- 有些知识工作更像“持续积累和编译知识”，不是“每次从零检索”
- 对中小规模知识库，先维护高质量摘要层和 wiki 层，可能比一上来就搭完整 embedding 检索链更简单

但这不等于：

- `RAG 已经过时`
- `以后都不需要检索`
- `长上下文一定比检索更好`

更准确的判断是：

1. 如果资料量不大、结构清晰、强调持续沉淀观点与总结，`LLM Wiki` 值得考虑
2. 如果问题强依赖最新规则、权限边界、来源可追溯，`RAG / file_search / metadata filtering` 仍然非常重要
3. 如果资料能一次性安全塞进上下文，`长上下文直塞资料` 往往比先搭 RAG 更简单
4. 如果资料很多、更新频繁、还要按地区、版本、业务线过滤，还是要靠检索系统

一句话记住：

`2026 更准确的说法不是“RAG 没用了”，而是“知识增强不只一种架构，RAG 只是其中一种基础路线”。`

#### 3.12.1 RAG 和 Wiki 的真正差别，不是“搜不搜”

很多人第一次听 `LLM Wiki` 时，会误以为：

`Wiki = 以后回答问题都不用搜了`

这不准确。

更准确的理解是：

- `RAG` 往往是 `现查原始资料`
- `Wiki` 往往是 `先把原始资料整理成知识页，再查知识页`

所以很多时候，两者在回答问题时都可能会搜索。

真正的差别是：

1. `搜的对象不同`
- `RAG` 搜的是原始 SOP、FAQ、规则原文、文档 chunk
- `Wiki` 搜的是整理后的专题页、总览页、差异页、术语页

2. `整理动作发生的时点不同`
- `RAG` 更像用户提问时临时查原文
- `Wiki` 更像在用户提问前，先把知识编译成一套手册

3. `系统重点不同`
- `RAG` 重点是检索原文准不准、召回全不全
- `Wiki` 重点是知识整理得好不好、摘要有没有失真

一句话压缩：

`RAG = 现查原文；Wiki = 先写手册，再查手册。`

#### 3.12.2 Wiki 整理了知识，不等于知识已经“长进模型脑子里”

这也是最容易误解的地方。

你可以先把知识系统分成 3 层：

1. `模型参数`
模型训练时学到的通用知识

2. `外部知识层`
你们自己维护的原始资料、wiki 页面、知识页

3. `运行时上下文`
每次回答时，系统实际喂给模型看的内容

所以 `Wiki` 的意思通常不是：

`模型已经永久记住这些知识，以后完全不用看资料。`

而是：

`你先把原始资料整理成更好用的外部知识页，模型回答时优先读取这些知识页。`

#### 3.12.3 为什么整理成 Wiki 后会明显更好用

原始 SOP 往往不是按“快速回答问题”写的，而是按“记录规则”写的。

所以原始资料常见问题是：

- 信息分散在多份文档里
- 一份讲标准流程，一份讲例外，一份讲地区差异
- 术语不统一
- 每次回答都要重新拼接

而整理成 `Wiki` 之后，优势在于：

1. `更容易理解`
原始文档是材料，wiki 是按问题组织过的知识页

2. `更容易复用`
高频问题不需要每次都重新检索、重新总结

3. `更容易做跨文档整合`
可以把 FAQ、SOP、区域规则整理成同一页总览

4. `更适合长期沉淀团队知识`
不仅模型能用，新人、运营、客服也能直接用

但它也有代价：

- 整理错了会把错误放大
- 摘要过度可能丢细节
- 最新规则变化后，wiki 可能还没同步

所以高风险场景下，`Wiki` 往往不能完全替代原始资料。

#### 3.12.4 一个物流例子：为什么原始 SOP 难用，整理成 Wiki 后明显更好用

假设原始资料散在 4 份文档里：

- `SOP-回单上传流程-v3`
- `FAQ-回单异常处理`
- `华南区域运营补充规则`
- `异常工单升级标准`

用户问的是：

`回单上传超时怎么办？`

如果走 `RAG`，系统通常会：

1. 从这 4 份原始资料里检索相关 chunk
2. 找到“签收后 2 小时内上传”“图片模糊要补拍”“华南区先核验司机”“超过 24 小时转异常工单”这些散落片段
3. 再把这些片段拼起来回答

如果平时先整理出一页 wiki：

`回单上传超时处理`

这页就可以直接按问题写成：

- 什么叫超时
- 常见原因
- 标准处理步骤
- 区域差异
- 什么时候转人工
- 对应来源文档

这时回答同样问题时，系统更像是：

1. 先查 `回单上传超时处理` 这页知识页
2. 直接基于这页回答
3. 只有在涉及最新版本、争议条款、精确责任时，再回查原始规则

一句话记住这个例子：

`原始 SOP 是材料仓库，Wiki 是问题导向的使用手册。`

#### 3.12.5 最现实的落地，不是二选一，而是 Wiki + RAG

很多真实产品不是只做 `RAG`，也不是只做 `Wiki`，而是：

1. `raw sources`
保留原始 SOP、FAQ、规则、政策文档

2. `wiki layer`
把高频主题整理成总览页、差异页、术语页

3. `answering layer`
- 普通问题优先查 wiki
- 最新规则、精确条款、可追溯答案再回查原文

你可以把它压缩成一句：

`Wiki 负责让知识更好用，RAG 负责让答案更可追溯。`

### 3.13 纯 LLM、Prompt 塞规则、长上下文、RAG、LLM Wiki、Agent 的差别

| 方式 | 它怎么工作 | 优点 | 风险 | 适用场景 |
|---|---|---|---|---|
| 纯 LLM | 靠模型已有知识回答 | 简单、快 | 容易泛化、幻觉 | 通用问答 |
| Prompt 塞规则 | 把少量业务规则直接写进 Prompt | 成本低、适合 Demo | 规则多了难维护 | 小场景、固定任务 |
| 长上下文直塞资料 | 把相关资料整段放进上下文，再一起回答 | 起步快、实现简单 | 成本高，资料变多后管理困难 | 小规模资料分析、一次性任务 |
| RAG | 先检索资料，再基于资料回答 | 更贴业务、可更新、可控性强 | 检索不到就答不好 | 知识问答、SOP 助手、规则问答 |
| LLM Wiki | 先把原始资料持续整理成 wiki，再基于 wiki 作答 | 知识会积累，摘要和交叉引用可复用 | wiki 质量会影响答案，可能偏离原始资料 | 长期研究、团队知识沉淀、个人知识库 |
| Agent | 多步调用工具完成任务 | 能真正办事 | 复杂度高 | 自动化流程、任务执行 |

### 3.14 RAG 常见 badcase

最常见的 4 类问题：

1. `知识没收全`
源文档本身就不完整

2. `切块切坏了`
一条规则被切散，导致上下文丢失

3. `检索召回不准`
该找的没找出来，不该找的找出来了

4. `生成阶段越权发挥`
明明资料里没有，模型还补充解释

## 4. 官方资料与阅读重点

如果你想把概念和最新官方语境对齐，优先看这 3 类资料就够用了。

### 官方资料 1：Retrieval

建议阅读：
- [OpenAI Retrieval Guide](https://developers.openai.com/api/docs/guides/retrieval)

重点看：
- retrieval 到底解决什么问题
- vector store 是什么
- semantic search 是怎么工作的
- 默认 chunking 是什么

补充记忆点：
- Retrieval guide 里提到，默认 chunk 大小是 `800 tokens`，chunk overlap 是 `400 tokens`
- 这说明“切块策略”不是 RAG 的细枝末节，而是核心质量变量

看完最该带走的核心结论：
- `Retrieval` 主要解决的是“企业知识和最新资料怎么被拿来回答”，不是让模型突然变得更会推理
- 一个 RAG 系统的质量，很多时候先输在 `切块 / 召回 / 精度`，还没轮到生成阶段
- `semantic search` 的核心不是按关键词硬匹配，而是按“语义是否相近”去找资料，所以用户问法和文档写法不一样时，系统仍有机会召回相关内容
- `default chunking` 可以先理解成“系统默认怎么切文档”：当前官方文档写的是默认按 `800 tokens` 一块来切，并在相邻块之间保留 `400 tokens overlap`，它的意义是既保证检索粒度，又尽量避免上下文被切断
- `vector store` 不是抽象概念，而是实际承载 chunk、embedding 和 metadata 的检索层
- 从产品视角看，RAG 质量问题要优先排查“资料有没有找对”，不要一上来怪模型

### 官方资料 2：File Search

建议阅读：
- [OpenAI File Search Guide](https://developers.openai.com/api/docs/guides/tools-file-search)

重点看：
- file search 怎么接到 Responses API
- vector store 怎么用
- 怎么限制结果数
- 怎么加 metadata 过滤

你只要先抓住这 3 个结论：
- `RAG 不一定要自己手搓检索链路`
- `file_search` 已经能做很多基础知识库问答能力
- metadata filtering 对业务场景特别重要

看完最该带走的核心结论：
- 很多知识问答产品，不需要从零自建整条检索链，可以先用 `file_search` 快速验证
- `metadata filtering` 在真实业务里很关键，因为很多问题天然带有 `地区 / 版本 / 生效时间 / 业务线` 这些过滤条件
- 真正的产品差异，不一定在“有没有检索”，而在“有没有把知识源、标签、过滤和边界设计好”
- 你要开始形成一个习惯：只问“能不能搜到”不够，还要问“能不能搜得对、搜得稳、搜得可控”

### 官方资料 3：Embeddings

建议阅读：
- [OpenAI Embeddings Guide](https://developers.openai.com/api/docs/guides/embeddings)
- [OpenAI Embeddings API Reference](https://platform.openai.com/docs/api-reference/embeddings/object?lang=node.js)

重点看：
- embedding 是什么
- 为什么语义检索离不开 embedding
- embedding 在搜索里的作用到底是什么

一句话记住：

`RAG 的底层不是“模型自己懂了文档”，而是“文档先被表示成可检索的向量，再被检索回来给模型看”。`

看完最该带走的核心结论：
- `embedding` 的价值不在“存知识”，而在“把文本变成可比较语义相似度的表示”
- 它解决的是“用户说法和文档写法不完全一样，但系统仍要把相关资料找出来”
- 只有 embedding 还不够，真实产品里还需要 `chunking + metadata + 路由规则` 一起配合
- 从产品角度记住一句就够：`embedding 是语义检索的底层表示，不是模型记住了文档`

### 行业补充：LLM Wiki（2026-04-04）

这不是官方文档，但作为 `2026 年 4 月` 很热的行业讨论，你最好知道它在说什么。

建议阅读：
- [Karpathy: LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)

重点看：
- `raw sources / wiki / schema` 三层架构
- 为什么 moderate scale 时，`index.md` 就可能先够用
- 为什么规模继续增长后，还是会需要搜索工具

你要得出的结论不是：

`RAG 死了`

而是：

`知识增强开始分化成多种路线：长上下文、RAG、LLM Wiki 都有各自边界。`

补一层现实判断：

- 截至 `2026-04-08`，OpenAI 仍然在提供并维护 `Retrieval` 和 `File Search`
- 这说明检索式知识增强并没有过时，只是已经不再是唯一解
- 你做产品判断时，应该比较的是 `哪种知识层最适合这个任务`，而不是默认所有问题都上 RAG

看完最该带走的核心结论：
- `Wiki` 的关键价值不是“以后不用搜”，而是“先把难用的原始资料整理成更好用的知识页”
- `Wiki` 不等于知识已经长进模型参数里，它通常仍然是外部知识层，只是比原始文档更适合被复用
- `RAG` 和 `Wiki` 的真正差别，不是“搜不搜”，而是“搜原文”还是“搜整理后的手册”
- 很多真实产品最好的落地不是二选一，而是 `Wiki 负责提高可用性，RAG 负责追溯原文和最新规则`

### 4.1 这一节你最终要记住什么

如果你只记 5 句话，就记这 5 句：

1. `Retrieval` 解决的是“怎么把企业资料拿给模型用”，不是让模型自己长出业务知识
2. `File Search` 说明很多 RAG 产品不必一开始就自建复杂检索链
3. `Embedding` 解决的是语义相似检索，不是“模型已经记住文档”
4. `Wiki` 解决的是知识整理与复用，不是宣布检索失效
5. 做 Day 2 最重要的不是会背术语，而是会判断：`这个问题该查原文，还是该先整理成知识页`

## 5. 三个练习

这 3 个练习依然尽量围绕你熟悉的 `网络货运 / 订单 / SOP / FAQ` 场景来做。

### 练习 1：判断一个问题该选哪种知识增强路线

#### 练习内容

判断下面这些任务，哪种路线更合适：

- `纯 LLM`
- `Prompt 塞规则`
- `长上下文直塞资料`
- `RAG / file_search`
- `LLM Wiki`
- `Agent（必要时可叠加 RAG）`

1. 用户问：`什么是 Prompt？`
2. 用户问：`平台回单上传超时后应该怎么处理？`
3. 用户问：`帮我想一个物流品牌 slogan`
4. 用户问：`华南区域 2026 年最新赔付规则是什么？`
5. 用户问：`把这段订单评价归类为哪种问题`
6. 用户问：`把最近 6 份赔付规则更新记录整理成差异摘要`
7. 运营同学要维护一份 `网络货运异常处理知识手册`，希望能持续沉淀 FAQ、规则差异和术语解释
8. 用户发起异常工单后，系统需要 `查订单状态 + 查规则 + 生成处理建议 + 发起流转`

##### 你要观察什么

- [ ] 这个问题是否依赖企业自己的知识
- [ ] 这个知识是否会更新
- [ ] 这个任务是否必须引用资料
- [ ] 这个任务是否可以只靠 Prompt 规则解决
- [ ] 这是一次性分析，还是长期积累知识
- [ ] 这个任务只是回答，还是还要真正执行动作

---

#### 笔记：什么任务适合哪种路线

判断一个任务该选哪种方案，可以先问 6 个问题：

1. 这个回答是不是依赖企业自己的资料？
2. 这份资料会不会经常更新？
3. 如果答错，风险是不是主要来自“知识不对”而不是“表达不好”？
4. 这个任务是不是“查资料后再回答”比“直接生成”更合理？
5. 这些资料是不是一次性就能安全放进上下文里？
6. 这个任务是不是还要调用系统、执行流程或跨多步判断？

你可以先按这个顺序判断：

1. 如果只是通用常识或创意生成，先想 `纯 LLM`
2. 如果规则很少且稳定，先想 `Prompt 塞规则`
3. 如果资料量不大，而且主要是一次性汇总比较，先想 `长上下文`
4. 如果问题依赖最新业务知识、还要按地区版本过滤，优先想 `RAG`
5. 如果目标是长期沉淀和复用知识，再考虑 `LLM Wiki`
6. 如果系统不仅要回答，还要去查单、提单、流转，就进入 `Agent`

---

#### 标准答案

更适合 `纯 LLM` 的：
- `什么是 Prompt？`
- `帮我想一个物流品牌 slogan`

更适合 `RAG / file_search` 的：
- `平台回单上传超时后应该怎么处理？`
- `华南区域 2026 年最新赔付规则是什么？`

更适合 `Prompt 塞规则` 的：
- `把这段订单评价归类为哪种问题`

更适合 `长上下文直塞资料` 的：
- `把最近 6 份赔付规则更新记录整理成差异摘要`

更适合 `LLM Wiki` 的：
- `持续维护网络货运异常处理知识手册`

更适合 `Agent` 的：
- `查订单状态 + 查规则 + 生成处理建议 + 发起流转`

原因可以压缩成一句：

`不是所有知识问题都默认上 RAG，你要先判断它是“直接回答”“一次性汇总”“长期沉淀知识”还是“真正执行任务”。`

---

#### 核心收获 / 方法论

`2026 年更重要的能力，不只是判断“要不要 RAG”，而是判断“该选哪种知识增强架构”。`

---

### 练习 2：设计一个最小可用物流知识层

#### 练习内容

假设你要做一个 `网络货运平台知识助手`，回答这几类问题：

- 回单什么时候上传
- 签收异常怎么处理
- 赔付规则怎么查
- 不同区域规则有什么区别

请你设计这套知识系统至少要有哪些层，以及每层放什么内容。

##### 你要观察什么

- [ ] 文档是否覆盖主要业务问题
- [ ] metadata 是否能支持过滤
- [ ] 文档是否区分版本、地区、规则类型
- [ ] 是否区分原始资料层和整理后的 wiki 层
- [ ] 是否想清楚哪类问题走长上下文，哪类问题走检索

---

#### 笔记：知识层不是“把文档全丢进去”

一个可用知识系统，至少可以先想 4 层：

1. `raw sources`
到底收哪些文档：
- FAQ
- SOP
- 赔付规则
- 操作手册
- 区域政策
- 工单复盘 / 运营口径 / 培训材料

2. `wiki / summary layer`
把高频主题沉淀成更稳定、可复用的知识页：
- 回单处理知识页
- 签收异常处理知识页
- 区域赔付差异对比页
- 常见术语解释页

3. `结构化标签`
至少考虑：
- 文档类型
- 业务线
- 地区
- 更新时间
- 规则主题
- 生效时间
- 来源文档 ID

4. `更新机制与路由规则`
如果规则更新了：
- 谁负责更新
- 更新后旧版本怎么办
- 是否保留版本号

如果用户提问时：
- 查 `最新规则 / 区域差异 / 可追溯答案`，优先走 `RAG`
- 做 `少量文档比对总结`，可以先走 `长上下文`
- 查 `稳定概念 / 已整理知识页`，可以优先走 `wiki`

一句话记住：

`知识产品不是“存文件”，而是“同时管理 raw sources、wiki、schema 和 retrieval 路由”。`

---

#### 标准答案

最小 raw sources 建议：
- `回单上传 FAQ`
- `签收异常处理 SOP`
- `赔付规则说明`
- `区域差异说明`
- `平台操作手册`
- `典型异常工单复盘`

最小 wiki layer 建议：
- `回单专题页`
- `签收异常处理总览`
- `赔付规则总览`
- `华南 / 华东差异对比页`
- `异常类型术语表`

建议的 metadata：
- `category`：FAQ / SOP / 规则 / 手册
- `topic`：回单 / 签收 / 赔付 / 状态同步
- `region`：全国 / 华南 / 华东
- `updated_at`：更新时间
- `version`：规则版本
- `effective_from`：生效时间
- `source_id`：来源文档编号

建议的最小路由：
- `最新规则查询` -> `RAG / file_search`
- `少量资料对比总结` -> `长上下文`
- `稳定知识解释` -> `wiki`

---

#### 核心收获 / 方法论

`知识增强系统先决定“知识怎么分层”，再决定“答案怎么生成”。`

### 练习 3：写两个最小可用的知识增强 Prompt

#### 练习内容

目标：
- 让模型基于“已检索到的资料”回答
- 资料不支持时，不要乱编
- 尽量指出答案来自哪类资料
- 区分“检索型问答”和“汇总型问答”的 Prompt 写法

##### 第一版 Prompt

```text
你是网络货运平台知识助手。

请根据给定资料回答用户问题。

如果资料中没有答案，请明确说明资料不足，不要编造。

资料：
{{retrieved_context}}

问题：
{{user_query}}
```

##### 你要观察什么

- [ ] 会不会脱离资料自由发挥
- [ ] 资料不足时会不会硬答
- [ ] 回答有没有把资料和结论绑定起来
- [ ] 遇到多份资料冲突时，会不会提示冲突而不是强行下结论

---

#### 笔记：不同知识路线，Prompt 的边界也不同

`RAG 型 Prompt` 最关键的，不是让模型“更会说”，而是让模型：

1. 只基于资料回答
2. 资料不够时明确说不够
3. 不要把常识脑补成企业规则
4. 最好能标出答案来自哪类资料

`长上下文 / wiki 型 Prompt` 更强调：

1. 先做归纳，不要遗漏冲突点
2. 明确区分“资料原文”与“你的总结”
3. 如果不同资料口径不一致，要直接指出

一句话记住：

`知识增强 Prompt 的第一职责，不是“多会说”，而是“守住对应知识层的边界”。`

---

#### 标准答案

`版本 A：检索型问答 Prompt`

```text
角色：
你是网络货运平台知识助手。

任务：
请仅基于给定资料回答用户问题，不要使用资料外的假设补充企业规则。

回答要求：
1. 优先基于资料直接回答
2. 如果资料中信息不足，明确回答“当前资料不足，无法确认”
3. 不要编造规则、时间、责任方或处理结论
4. 如果资料中存在区域、版本或条件差异，回答时要指出
5. 回答尽量简洁，并说明答案主要依据的是哪类资料

资料：
{{retrieved_context}}

问题：
{{user_query}}
```

`版本 B：汇总 / wiki 型 Prompt`

```text
角色：
你是网络货运平台知识整理助手。

任务：
请基于给定资料，对问题做总结、对比和归纳。

回答要求：
1. 先给出结论摘要，再给出依据
2. 如果不同资料存在冲突，单独列出“冲突点”
3. 不要把资料中没有的企业规则补充进去
4. 明确区分“资料原文信息”和“你的归纳总结”
5. 如果资料不足以支持明确结论，直接说明“当前资料不足，暂不能下结论”

资料：
{{source_pack}}

问题：
{{user_query}}
```

---

#### 核心收获 / 方法论

`先分清你面对的是“检索型任务”还是“汇总型任务”，再写 Prompt，效果会比统一套一个模板更稳。`
