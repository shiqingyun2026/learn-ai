## 4. 核心概念

### 4.1 Prompt 是什么

你可以先这样理解：

`Prompt 不是“随便问一句话”，而是你给模型下达任务、约束、目标和输出格式的一整套指令。`

对产品经理来说，Prompt 最重要的不是写得多花，而是：
- [ ] 任务是否明确
- [ ] 角色是否明确
- [ ] 输出格式是否明确
- [ ] 边界条件是否明确

### 4.2 System Prompt 是什么

你可以先这样理解：

`System Prompt 是最高优先级的行为约束，用来定义模型是谁、要做什么、不能做什么、输出应该长什么样。`

你可以把它和 `User Prompt` 对比着记：
- `User Prompt` 更像：“这次让我干什么”
- `System Prompt` 更像：“你平时必须按什么原则干活”

对你未来做 AI 产品很重要，因为它决定：
- [ ] 模型扮演什么角色
- [ ] 模型回答的边界在哪里
- [ ] 模型要遵守什么业务规则

#### 补充笔记：行为约束和边界设定的差别

可以先粗暴地记成：
- `行为约束`：规定“怎么做”
- `边界设定`：规定“做到哪儿为止”

`行为约束` 重点是模型在执行任务时要遵守的规则，比如：
- 不要编造信息
- 语气要简洁
- 信息不足时要明确说明
- 只能输出 JSON
- 不要输出无关内容

它管的是：
`模型做事的方式`

所以它回答的是：
`你可以做这个任务，但你要按这些规矩做。`

`边界设定` 重点是模型能做什么、不能做什么，职责范围到哪里，比如：
- 只处理订单异常，不处理赔付裁定
- 只能做初步归因，不能做最终责任判定
- 遇到高风险场景必须转人工
- 不能调用未授权系统
- 不回答与货运业务无关的问题

它管的是：
`模型职责的范围`

所以它回答的是：
`哪些事归你做，哪些事不归你做。`

#### 2026 补充：`System Prompt` 这个概念没过时，但 API 语境变了

截至 `2026-04-08`，如果你看最新官方文档，会同时看到这些词：

- `system message`
- `developer message`
- `instructions`

你可以先这样理解：

- `System Prompt` 仍然是最适合入门记忆的概念
- 真到工程实现里，研发同学现在更常会提 `Responses API`
- 在 `Responses API` 里，系统级约束常通过 `instructions` 传入
- 在新的 Prompt engineering 文档里，也经常把规则和业务逻辑写在 `developer message`

对你最有用的一句话是：

`Day 2 继续学 System Prompt 没问题，但跟研发沟通时，要知道它在最新接口里常对应 instructions 或 developer message。`

### 4.3 Few-shot 是什么

Few-shot 的本质不是“给模型喂知识”，而是：

`给模型看几个你想要的输入输出样例，让它学会你要的回答模式。`

一句话记忆：

`few-shot = 用少量范例教模型当场学会你的做题格式。`

适合 few-shot 的情况：
- [ ] 你希望分类口径统一
- [ ] 你希望输出格式稳定
- [ ] 任务规则不复杂，但语言表达容易漂

不一定要 few-shot 的情况：
- [ ] 任务很简单
- [ ] 规则非常明确
- [ ] 结构化输出已经足够约束模型

few-shot 解决的是：
`怎么做得更像你想要的样子`

#### 补充笔记：什么叫“语言表达容易漂”

这里的“语言表达容易漂”，意思是：

`同一个任务，模型每次都可能用不同说法表达，导致输出不够稳定。`

从“语义”上看可能差不多，但从“产品可用性”看会很麻烦，因为：
- 前端不好接
- 数据不好统计
- 规则不好判断
- 结果口径不统一

#### 补充笔记：为什么大模型输出容易“飘”

本质上是因为它是在按概率生成下一个 token，而不是像规则引擎那样按固定逻辑返回唯一答案。

你可以先记两层原因：

1. `它是概率模型`
同一个输入下，往后接的词往往不止一个“合理答案”。
模型会在多个高概率候选里继续生成，所以表达天然会有波动。

2. `它是 LLM`
LLM 的目标通常不是“永远用同一句话回答”，而是：
`生成在语言上自然、连贯、统计上合理的文本。`

这就意味着，它更擅长“像人一样说话”，不擅长“像程序一样永远一模一样”。

所以常见情况是：
- 语义可能接近
- 形式却不稳定

为什么产品里这会变成问题？

因为产品需要的是：
- 可解析
- 可统计
- 可复用
- 可接后续流程

而 LLM 天然偏向的是：
- 自然表达
- 灵活表述
- 上下文适配
- 多样化生成

这两者之间有张力，所以才需要：
- `System Prompt`
- `few-shot`
- `结构化输出`
- `tool calling`
- `规则兜底`

你也可以这样记：

`LLM 天生追求“像人”，产品系统追求“像机器”。我们做 AI 产品，本质上就是在这两者之间加约束，把“会说话”改造成“可稳定交付”。`



### 4.4 结构化输出为什么重要

这是你今天最该吃透的点。

自然语言输出的问题是：
- [ ] 不稳定
- [ ] 字段不固定
- [ ] 难以直接接前端和系统
- [ ] 难以评测和统计

结构化输出的价值是：
- [ ] 可解析
- [ ] 可校验
- [ ] 可统计
- [ ] 可作为后续流程输入

对 AI 产品经理来说，结构化输出意味着：

`模型不只是“会说话”，而是能把结果交给产品流程继续处理。`

#### 2026 补充：优先记 `Structured Outputs`，不要只停留在 `JSON mode`

现在更准确的理解是：

- `JSON mode` 只能尽量保证返回的是合法 JSON
- `Structured Outputs` 才能真正要求模型按你定义的 schema 交付
- 如果你的目标是“把模型结果接进产品流程”，应该优先想 `schema`

一句话记住：

`JSON mode 解决“像不像 JSON”，Structured Outputs 解决“是不是我定义的那份接口”。`

#### 2026 补充：什么时候用 `text.format`，什么时候用 `function calling`

你可以先这样分：

- 如果你只是想让模型对用户输出固定结构，优先用 `Structured Outputs`
- 如果你是想让模型去调用系统能力、函数、工具，那就进入 `function calling`

所以 Day 2 的一个新记忆点是：

`结构化输出负责“交付格式”，tool / function calling 负责“能力连接”。`

### 4.5 2026 版 Day 2 你要知道的 4 个实现变化

这 4 个点会让你的 Day 2 不至于学成“旧时代 Prompt”：

1. `Responses API` 是新项目默认优先考虑的接口
以前很多教程会从 `Chat Completions` 讲起，但现在做新项目，应该先知道 `Responses API`。

2. `System Prompt` 的实现入口更灵活了
除了传统消息里的 system role，你还会看到 `instructions` 和 `developer message`。

3. `Structured Outputs` 比 “只要求返回 JSON” 更重要
做产品要优先考虑 schema，而不是只写一句“请输出 JSON”。

4. Prompt 已经开始进入 `版本化 + 评测 + 优化` 阶段
现在不只是“写提示词”，还要学会把 Prompt 当成可复用资产来管理、评测和迭代。

#### 补充笔记：传统模型和 LLM 的差别

你可以先记成一句：

`传统模型负责“算”，LLM 负责“懂”和“说”，两者可以配合。`

更具体一点：
- `传统模型` 适合明确、单点、稳定的预测判断
- `LLM` 适合理解语言、处理复杂上下文、生成结果
- `LLM` 很强，但不代表能替代所有传统模型

一句话压缩：

`传统算法模型更擅长“把一个问题算准”，LLM 更擅长“把一个复杂语言任务理解并生成结果”。`

#### 补充笔记：Schema 是什么

你可以先把 `schema` 理解成：

`一份数据格式合同`

从产品经理视角看，它很像：

`AI 输出字段的接口定义`

它规定了：
- 必须有哪些字段
- 每个字段叫什么
- 每个字段是什么类型
- 哪些值是允许的
- 哪些字段不能缺

#### 补充笔记：Schema 有什么用

最核心的用处有 4 个：

1. `让输出稳定`
模型不能今天叫 `type`，明天叫 `category`

2. `让系统能接`
前端、后端、工作流都能放心读取固定字段

3. `让数据能统计`
比如你能统计：
- 各类异常占比
- 人工复核率
- 平均 `confidence`

4. `减少模型乱发挥`
比如 `category` 只能从枚举里选，不能自己发明一个“司机侧履约意愿不足”

### 5.1 延伸预习：RAG 是什么

这部分是 `Day 3` 的内容，这里先建立一个直觉即可。

你可以先把 `RAG` 记成一句：

`RAG = 先查资料，再让大模型基于查到的资料回答。`

RAG 的价值不是让模型更聪明，而是：

`让模型回答时更贴近你自己的知识和规则。`

如果 `没有 RAG`，模型只能靠它训练时学到的通用知识来回答。

如果 `接了 RAG`，它更像回答前先翻了你们公司的制度手册、FAQ、SOP，再给你答复。

最基础的流程是：
1. 用户提问
2. 系统先去知识库里找相关资料
3. 找到你们平台的异常处理 SOP、取消发货规则、赔付说明
4. 再把这些资料连同问题一起交给模型
5. 模型基于这些资料回答

#### 一个简化对比

| 方式 | 怎么回答 | 优点 | 风险 | 适用场景 |
|---|---|---|---|---|
| 纯 LLM | 靠模型已有知识回答 | 快、简单 | 容易泛化、幻觉 | 通用问答、灵感生成 |
| Prompt 塞规则 | 把少量规则直接写进提示词 | 适合小规模场景 | 规则多了难维护 | 小型 Demo、固定任务 |
| RAG | 先查知识库再回答 | 更贴业务、易更新 | 检索不到就答不好 | SOP、客服、知识助手 |
| Agent | 多步调工具完成任务 | 能真正办事 | 复杂度更高 | 自动化流程、任务执行 |

### 练习 1：评价主题分类 Prompt

任务目标：
- 把一段用户评价分到固定主题里

分类标签先固定成这 6 类：
- `正向体验`
- `时效问题`
- `服务态度问题`
- `货损货差问题`
- `信息同步问题`
- `其他`

#### 练习内容

##### 第一版 Prompt

```text
你是网络货运平台的订单评价分类助手。

你的任务是根据用户输入的订单评价，将其归类到以下标签之一：
正向体验、时效问题、服务态度问题、货损货差问题、信息同步问题、其他。

要求：
1. 只能返回一个标签
2. 不要解释原因
3. 如果信息不足，返回“其他”
4. 如果评价同时包含多个问题，请返回最主要的问题标签

订单评价：
[在这里填入评价文本]
```

##### 你要观察什么

- [ ] 是否总能只输出一个标签
- [ ] 是否会输出解释
- [ ] 是否会分类漂移
- [ ] 面对一条评价里有多个问题时，主标签是否稳定

##### 迭代方向

如果输出不稳定，就补：
- [ ] 反例
- [ ] few-shot 示例
- [ ] 更清楚的标签定义

#### 笔记：练习 1 的局部优化动作

先记住一句：

`Prompt 优化不是“文学润色”，而是像做产品规则一样，把模型最容易犯错的地方一个个堵上。`

##### 1. 重要信息需要重复强调

比如：
- `只能返回一个标签`
- `你的回答必须且只能是以下六个标签之一：正向体验、时效问题、服务态度问题、货损货差问题、信息同步问题、其他`

这两句看起来像在重复，但作用不一样：
- 前面第一次列标签，主要是在 `定义任务`
- 后面再列一次，主要是在 `约束输出`

你可以这样理解：
- 前一次是在说：`题目范围是什么`
- 后一次是在说：`答案只能这样交`

哪些信息值得冗余：
- `输出必须标准化`：模型最容易多说两句或换个表达
- `输出是封闭集合/枚举值`：模型容易自己发明一个差不多的新词
- `高风险边界`：一旦错，不是格式难看，而是业务风险
- `优先级规则`：模型在冲突场景里最容易摇摆
- `你已经发现模型爱“换说法”` 的位置

不太需要重复的内容：
- `背景介绍`
- `常识性说明`
- `模型已经稳定执行的规则`

一句口诀：

`背景少重复，规则可重复，输出约束要重复，高风险边界宁可多说一遍。`

##### 2. 增加边界规则

这条规则很关键：

```text
如果评价既提到“实际送达晚”，又提到“系统状态更新慢”，优先看用户抱怨重点：
- 重点在“货到得晚”时，返回“时效问题”
- 重点在“状态/回单/进度信息不准或不同步”时，返回“信息同步问题”
```

为什么要加？

因为如果只写标签定义，不写冲突处理规则，模型在边界样本上会经常漂。
而真实产品最难的，往往不是明显样本，而是：

`一条评价里同时有多个信号，主标签到底怎么定`

所以这是从普通 Prompt 走向“可用于产品”的关键升级：

`不只是描述标签，还定义标签冲突时怎么判。`

##### 3. 一个最小迭代动作

如果你刚开始练，可以先按这个最小动作走：

1. 先写基础版：`角色 + 任务 + 标签 + 输出要求`
2. 跑几条真实数据
3. 找最常错的地方：
   - 标签混淆
   - 输出不规范
   - 同义词乱飞
   - 多问题不会判主标签
4. 针对错误补规则
5. 再决定要不要加 `few-shot`

##### 4. Badcase 设计

做练习时，不要只看“它能不能答对”，还要预判：

`它会怎么错`

可以用这 4 步看每条评价：

1. 这条评价的理想标签是什么
2. 模型最可能误判成什么
3. 为什么容易误判
4. 如果真误判了，应该补哪类规则

样本不应该全靠拍脑袋设计，也不能只依赖真实数据。
更有效的是：

`真实数据打底，人工样本补边界，badcase 继续反哺。`

你后面如果做样本池，至少要有：
- `常规样本`
- `边界样本`
- `高风险样本`
- `反例样本`
- `低信息样本`
- `badcase 记录表`

##### 5. 什么样的样本最有价值

- `高频真实样本`：代表用户最常见的问题
- `边界样本`：最容易混淆标签
- `高风险样本`：错了代价大
- `反例样本`：长得像，但不该判成某类
- `低信息样本`：测试模型会不会乱猜

#### 核心收获 / 方法论：Prompt 最佳设计流程

围绕这次“订单评价分类”练习，真正要记住的不是某一版 Prompt 文案，而是：

`Prompt 的最佳设计流程 = 先定义任务和评测，再写 Prompt，再用 badcase 迭代。`

这一节是完整流程，范围比上面的“局部优化动作”更大。

##### 1. 先定义任务，不要一上来就写 Prompt

先想清楚这 4 件事：
- 这个能力要解决什么问题
- 输入是什么
- 输出是什么
- 输出结果后面要接什么流程

在这次练习里，真实任务不是“随便分析评价”，而是：

`把订单评价稳定分到固定标签里，供后续统计、回访、质检、运营分析使用。`

##### 2. 先定义输出合同，再写提示词

在写 Prompt 前，先把输出定死：
- 只能返回一个标签
- 标签只能从固定 6 类中选择
- 不允许解释
- 不允许发明新词

你可以把这一步理解成：

`先定接口，再写提示词`

##### 3. 先定评测标准，否则优化会变成凭感觉

这次练习里，至少要看这几项：
- 分类是否正确
- 是否只返回一个标签
- 是否输出解释或额外文字
- 多问题评价下主标签是否稳定
- `时效问题` 和 `信息同步问题` 是否会混淆

没有评测标准，Prompt 优化很容易变成“我觉得这样写更顺”，但这不是产品优化。

##### 4. 先准备样本，再开始迭代

这次练习的样本其实应该分成 3 层：
- `常规样本`：明显属于某一类的评价
- `边界样本`：最容易混淆标签的评价
- `高风险样本`：错了会直接影响业务判断的评价

样本来源的最佳组合是：

`真实数据打底，人工样本补边界，badcase 继续反哺。`

##### 5. 基础版 Prompt 只要够用，不要一开始写太重

基础版只需要 4 块：
- 角色
- 任务
- 标签定义
- 输出要求

也就是你一开始写的那类版本：

`角色 + 任务 + 标签说明 + 只能返回一个标签`

先跑起来，再根据错误补规则，而不是一开始就堆很多字。

##### 6. Prompt 优化的核心不是“润色”，而是“堵错误”

这次练习里，我们真正做的优化有 5 类：

1. `补标签定义`
比如把“信息同步问题”定义得更贴真实业务语言

2. `补优先级规则`
比如“前半句礼貌性肯定，后半句明确投诉，优先按投诉内容分类”

3. `补冲突处理规则`
比如“时效问题”和“信息同步问题”同时出现时，按用户抱怨重点判”

4. `强化输出约束`
比如“你的回答必须且只能是以下六个标签之一”

5. `补 few-shot`
专门补最容易混淆的边界样本，而不是补那些本来就很明显的样本

所以，Prompt 优化不是“文学润色”，而是：

`像做产品规则一样，把模型最容易犯错的地方一个个堵上。`

##### 7. Few-shot 不是随便加，而是专门补边界

这次练习里，few-shot 最该补的不是“明显正确”的样本，而是：
- `正向体验` vs `其他`
- `时效问题` vs `信息同步问题`
- `礼貌性肯定 + 明确投诉`

一句话记住：

`few-shot 不是为了让 Prompt 看起来更专业，而是为了让模型在最容易摇摆的地方更稳定。`

##### 8. Badcase 分析才是真正的 AI 产品工作

做练习时，不要只看“它答没答对”，还要看：

1. 这条评价的理想标签是什么
2. 模型最可能误判成什么
3. 为什么容易误判
4. 如果误判了，应该补哪类规则

这套方法和真实 AI 产品上线前做评测，其实是同一条逻辑。

##### 9. 这次练习可以沉淀成一个通用 SOP

以后你做分类类 Prompt，都可以按这条 SOP 走：

1. 定义任务目标
2. 定义输出合同
3. 定义评测标准
4. 准备真实样本和边界样本
5. 写基础版 Prompt
6. 跑测试，识别错误模式
7. 按错误补规则
8. 只在必要时补 few-shot
9. 稳定后冻结版本，继续收集 badcase

##### 10. 一句话总结

`Prompt 的最佳设计流程，不是“想到一句好提示词”，而是“把任务、输出、规则、样本、评测和 badcase 串成一个迭代闭环”。`

##### 第一版 Prompt


- [ ] 明确用户角色
- [ ] 明确禁止性要求
- [ ] 给 2 个好示例

---

#### 笔记：把建议类 Prompt 规则拆成 5 类

以后你写建议类 Prompt 的规则，可以先按这 5 类去检查：

##### 1. 输出闭集

我允许它输出哪些标签或字段值？

##### 2. 真实性与保守性

它在不确定时会不会乱编？要不要先核验？

##### 3. 建议质量

它给的是不是可执行动作，而不是空话？

##### 4. 业务判定

多问题、边界场景、优先级规则有没有写清楚？

##### 5. 交付格式

它最后输出的格式能不能直接用？

##### 用这版 Prompt 重新归类

我帮你直接归类：

`A. 输出闭集约束`
- 情感倾向必须且只能是 3 个标签之一
- 问题类型必须且只能是 5 个标签之一

`B. 真实性与保守性约束`
- 不要编造不存在的事实
- 如果信息不足，明确指出还需要补充哪些信息
- 涉及原因或责任判断时先核验

`C. 建议质量约束`
- 建议必须具体、可执行、不能空泛
- 优先写客服或运营下一步可立即执行的动作

`D. 业务判定规则`
- 正面评价 -> 其他
- 正面评价 -> 无需跟进
- 信息不足 -> 先记录
- 多问题 -> 取最主要
- 货损优先级更高
- 时效 vs 信息同步怎么判

`E. 输出格式约束`
- 不要输出额外解释
- 不要合并多条评价
- 按固定格式逐条输出

##### 最后记这个口诀

以后写 Prompt 的通用规则时，按这个顺序想最顺：

`先管输出能不能收住，再管会不会乱编，再管建议有没有用，再管业务口径稳不稳，最后管格式。`

也就是：

`收住输出 -> 防止乱编 -> 提高建议质量 -> 写清业务规则 -> 固定交付格式`

#### 标准答案：订单评价分析 Prompt

下面这版可以作为练习 2 的标准参考答案，后面的抽象模板也基于这版来提炼。

```text
角色：
你是网络货运平台的订单评价分析助手。

任务：
根据用户输入的订单评价，识别情感倾向、主要问题类型，并给客服或运营输出具体、可执行的跟进建议。

情感倾向标签定义：
- 正面：整体满意，主要在表扬服务、时效、沟通或整体体验，且没有明确负向投诉
- 负面：用户有明确抱怨、投诉或明显不满
- 中性：评价信息不足、表达模糊，或没有明确正负向倾向

问题类型标签定义：
- 时效问题：主要抱怨到达晚、送货慢、延误、等待时间过长
- 服务态度问题：主要抱怨司机或客服态度差、沟通差、不耐烦、敷衍、不专业、不配合
- 货损货差问题：主要抱怨货物破损、丢失、少货、错货、数量不对
- 信息同步问题：主要抱怨订单状态更新不及时、回单上传慢、进度信息缺失、不准确、系统显示与实际不一致
- 其他：以上都不明显，或信息不足，或无法判断主要问题

一、输出闭集约束
1. 情感倾向必须且只能是以下三个标签之一：正面、负面、中性
2. 问题类型必须且只能是以下五个标签之一：时效问题、服务态度问题、货损货差问题、信息同步问题、其他

二、真实性与保守性约束
3. 不要编造不存在的事实
4. 如果信息不足，明确指出还需要补充哪些信息
5. 如果建议涉及原因判断、责任判断或异常说明，优先先写“核验/确认”，不要默认直接解释原因或直接处理
6. 如果评价信息不足，优先输出“建议先记录，必要时再补充信息”，不要默认直接进入回访或升级处理流程

三、建议质量约束
7. 跟进建议必须具体、可执行、不能空泛
8. 跟进建议优先写客服或运营下一步可立即执行的动作
9. 不要输出“加强、优化、提升、持续关注”这类没有直接动作的空泛表述

四、业务判定规则
10. 如果评价明显正面且没有实质性投诉，问题类型返回“其他”
11. 如果是正面评价，跟进建议统一写为“无需跟进，建议记录为正向评价样本。”，待补充信息写“无”
12. 如果一条评价同时涉及多个问题，只返回最主要、优先级最高的问题类型
13. 如果同时涉及货损货差和其他问题，优先返回“货损货差问题”
14. 如果同时涉及“实际送达晚”和“系统状态更新慢”，优先按用户抱怨重点判断：
   - 重点在“货到得晚”时，返回“时效问题”
   - 重点在“状态/回单/进度信息不准或不同步”时，返回“信息同步问题”
15. 如果问题涉及投诉、货损、强负面体验或需要核验事实，优先给出“建议回访/核验/升级处理”的动作

五、输出格式约束
16. 请按“订单评价1、订单评价2……”逐条输出，不要合并分析
17. 每条评价都必须使用以下固定格式：
订单评价X：
·情感倾向：
·问题类型：
·跟进建议：
·待补充信息：
18. 不要输出额外解释，不要补充与固定格式无关的内容
```

#### 核心收获 / 方法论：从分类 Prompt 到建议生成 Prompt

练习 2 最重要的收获是：

`分类任务解决的是“这是什么问题”，建议生成任务解决的是“接下来怎么处理”。`

从 Prompt 设计角度看，升级点有 4 个：

1. `任务目标升级`
从“识别问题”升级成“给动作建议”

2. `输出结构升级`
从“单标签”升级成“情感倾向 + 问题类型 + 跟进建议 + 待补充信息”

3. `约束方式升级`
除了要求格式稳定，还要要求建议：
- 可执行
- 不空泛
- 不编造
- 不越权

4. `评测重点升级`
从看“分类准不准”，升级成看：
- 建议能不能直接用
- 是否符合业务角色
- 是否能区分回访和记录

一句话总结：

`分类 Prompt 更像判题，建议生成 Prompt 更像给业务动作。`

#### 可复用模板：建议类 Prompt 编写模板

这版不是针对订单评价本身，而是把练习 2 里已经验证过的方法，抽象成一个可复用模板。

```text
角色：
你是[业务场景]的[分析助手名称]。

任务：
根据用户输入的[文本类型]，识别[维度1]、主要问题类型，并给[使用对象]输出具体、可执行的跟进建议。

[维度1]标签定义：
- [标签A]：[定义]
- [标签B]：[定义]
- [标签C]：[定义]

问题类型标签定义：
- [问题类型1]：[定义]
- [问题类型2]：[定义]
- [问题类型3]：[定义]
- [问题类型4]：[定义]
- 其他：[兜底定义]

一、输出闭集约束
1. [维度1]必须且只能是以下标签之一：[标签A]、[标签B]、[标签C]
2. 问题类型必须且只能是以下标签之一：[问题类型1]、[问题类型2]、[问题类型3]、[问题类型4]、其他

二、真实性与保守性约束
3. 不要编造不存在的事实
4. 如果信息不足，明确指出还需要补充哪些信息
5. 如果建议涉及原因判断、责任判断或异常说明，优先先写“核验/确认”，不要默认直接解释原因或直接处理
6. 如果输入信息不足，优先输出“建议先记录，必要时再补充信息”，不要默认直接进入回访或升级处理流程

三、建议质量约束
7. 跟进建议必须具体、可执行、不能空泛
8. 跟进建议优先写[使用对象]下一步可立即执行的动作
9. 不要输出“加强、优化、提升、持续关注”这类没有直接动作的空泛表述

四、业务判定规则
10. 如果内容明显正面且没有实质性投诉，问题类型返回“其他”
11. 如果是正面内容，跟进建议统一写为“[正面场景固定建议]”，待补充信息写“无”
12. 如果一条内容同时涉及多个问题，只返回最主要、优先级最高的问题类型
13. 如果同时涉及[高优先级问题]和其他问题，优先返回“[高优先级问题]”
14. 如果同时涉及[易混淆问题A]和[易混淆问题B]，优先按用户抱怨重点判断：
   - 重点在[A判断口径]时，返回“[问题A]”
   - 重点在[B判断口径]时，返回“[问题B]”
15. 如果问题涉及投诉、高风险问题、强负面体验或需要核验事实，优先给出“建议回访/核验/升级处理”的动作

五、输出格式约束
16. 请按“[输入项1]、[输入项2]……”逐条输出，不要合并分析
17. 每条内容都必须使用以下固定格式：
[输入项X]：
·[维度1]：
·问题类型：
·跟进建议：
·待补充信息：
18. 不要输出额外解释，不要补充与固定格式无关的内容
```

##### 这个模板的骨架

1. `角色`
2. `任务`
3. `标签定义`
4. `五类规则`
5. `固定输出格式`

##### 以后写这类 Prompt，就按这 5 类检查

1. `输出闭集`
是不是先收住标签和字段值

2. `真实性与保守性`
信息不够时会不会乱编，要不要先核验

3. `建议质量`
给的是不是具体动作，而不是空话

4. `业务判定`
多问题、边界场景、优先级规则有没有写清楚

5. `交付格式`
最后输出能不能直接用

##### 最后记这个口诀

`收住输出 -> 防止乱编 -> 提高建议质量 -> 写清业务规则 -> 固定交付格式`

### 练习 3：结构化 schema 输出

任务目标：
- 把订单评价分析结果输出成固定字段，后续可直接接产品界面或流程

#### 练习内容

##### 第一版 Prompt

```text
你是网络货运平台的订单评价分析助手。

请根据输入内容，输出 JSON 结果。

字段要求：
- sentiment: 情绪倾向，只能是 positive、neutral、negative
- primary_issue: 主要问题分类，只能是正向体验、时效问题、服务态度问题、货损货差问题、信息同步问题、其他
- confidence: 0 到 1 的小数
- summary: 一句话总结
- follow_up_action: 下一步建议动作
- need_callback: true 或 false

要求：
1. 只输出 JSON
2. 不要输出 markdown
3. 不要补充额外说明

订单评价：
[在这里填入评价文本]
```

##### 你要观察什么

- [ ] 字段是否稳定
- [ ] 分类值是否跑出枚举
- [ ] confidence 是否乱给
- [ ] 是否能直接用于界面展示
- [ ] `need_callback` 是否符合业务直觉

##### 迭代方向

如果输出还不稳，就继续补：
- [ ] 明确每个字段的业务含义
- [ ] 给枚举字段写死可选值
- [ ] 给布尔字段写清触发规则
- [ ] 给信息不足场景准备兜底值
- [ ] 限制不能新增字段、不能漏字段

---

#### 笔记：练习 3 的方法要点

先记一句：

`结构化输出的重点，不是“让模型返回一段像 JSON 的文字”，而是先把输出当成接口，再让模型按接口交付。`

##### 1. 先定义字段，不要先写 Prompt

练习 3 和前两个练习最大的区别是：

`前两个练习更像“让模型回答”，练习 3 更像“让模型按固定接口交结果”。`

所以第一步不是想怎么提问，而是先想：
- 后面谁要用这个结果
- 前端要展示什么
- 运营要看什么
- 工作流要拿什么字段继续处理

这次这组字段，其实已经分别对应不同用途：
- `sentiment`：给情绪判断和统计
- `primary_issue`：给问题归类和报表
- `summary`：给人快速看懂
- `follow_up_action`：给客服或运营下一步动作
- `need_callback`：给流程判断要不要人工介入
- `confidence`：给质检或灰度观察参考

##### 2. 先区分“展示字段”和“流程字段”

不是所有字段都服务同一个目标。

你可以把它们分成两类：

- `展示字段`
  - `summary`
  - `follow_up_action`
- `流程字段`
  - `sentiment`
  - `primary_issue`
  - `need_callback`
  - `confidence`

为什么这一步重要？

因为很多初学者写结构化输出时，会把字段全写成“说明性文字”，结果看起来像结构化，实际上还是一段自然语言换了个壳。

真正产品可用的结构，至少要有一部分字段可以：
- 被程序直接判断
- 被前端直接渲染
- 被统计系统直接聚合

##### 3. 封闭字段一定要枚举化

像下面这些字段，就不能让模型自由发挥：
- `sentiment`
- `primary_issue`
- `need_callback`

因为这些字段后面通常会被：
- 前端控件读取
- 报表系统统计
- 工作流条件判断

所以要尽量写成：
- `sentiment` 只能是 `positive`、`neutral`、`negative`
- `primary_issue` 只能是 `正向体验`、`时效问题`、`服务态度问题`、`货损货差问题`、`信息同步问题`、`其他`
- `need_callback` 只能是 `true` 或 `false`

这一步本质上是在做：

`把模型输出从“开放表达”改成“封闭集合”。`

##### 4. 兜底逻辑要提前写，不要等模型自由发挥

结构化输出里，最容易翻车的不是“明显样本”，而是：
- 信息不够
- 同时出现多个问题
- 正向评价但又提了一点小瑕疵
- 需要不要回访不太明确

所以你要提前规定：
- 信息不足时，`primary_issue` 返回什么
- 正向评价时，`follow_up_action` 怎么写
- `need_callback` 在什么情况下必须是 `true`
- `missing_info` 是空数组还是补哪些信息

如果这些规则不提前写，模型就很容易：
- 有时返回空字符串
- 有时自己加新字段
- 有时用自然语言解释一大段

##### 5. `confidence` 不是装饰字段

很多人会机械地加一个 `confidence`，但没定义它到底表示什么。

如果你真的要这个字段，最好明确：

`confidence 表示模型对“本次情绪和主问题判断”的把握程度，不表示用户满意度，也不表示事件严重程度。`

否则后面就会出现两种常见误用：
- 把 `confidence` 当成“用户有多生气”
- 把 `confidence` 当成“问题有多严重”

这两个都不对。

##### 6. `confidence` 更像模型自评，不是严格概率

如果只是像练习 3 这样，在 Prompt 里要求模型输出一个 `confidence` 小数，那么这个值更接近：

`模型对自己这次判断稳不稳的主观表达`

它通常会参考这些信号：
- 文本信号是否明确
- 标签是否单一
- 信息是否充分
- 是否存在边界冲突

比如：
- `司机电话里态度很差，一直不耐烦` -> 往往更容易给高分
- `整体一般，说不上哪里不好` -> 往往更容易给低分

但这里要特别注意：

`模型自评 confidence 不等于经过校准的真实正确率。`

所以：
- `0.90` 不等于“90% 一定答对”
- `0.60` 也不等于“60% 会答对”

它更适合做：
- 粗分流
- 人工复核辅助
- badcase 观察

##### 7. 为什么很多产品早期更适合用“高 / 中 / 低”

早期产品常常没有足够数据去证明：
- `0.81` 和 `0.86` 是否真有稳定差异
- `0.72` 和 `0.68` 是否对应不同处理策略

所以很多时候，先落成：
- `高`
- `中`
- `低`

反而更稳。

这样做的好处是：
- 对业务同学更好理解
- 更容易直接设计分流规则
- 更不容易被“假精确的小数”误导
- 更适合早期样本量还不够时的阶段

一句话记住：

`产品早期更需要“可用的置信度分档”，而不是“看起来很精确的小数”。`

##### 8. 怎么用订单评价数据，手工做一个最简单的 confidence 校准表

你可以先用最小版本来做：

1. 准备一批人工标注好的订单评价样本
2. 跑模型，拿到：
   - 模型标签
   - 原始 `confidence`
3. 新增一列 `是否正确`
   - 预测正确填 `1`
   - 预测错误填 `0`
4. 把 `confidence` 分成 3 档：
   - `高`：`>= 0.85`
   - `中`：`0.70 - 0.84`
   - `低`：`< 0.70`
5. 统计每一档的：
   - 样本数
   - 正确数
   - 实际正确率

最后你会得到一张最简单的校准表，例如：

| 置信度档位 | 样本数 | 正确数 | 实际正确率 |
|---|---:|---:|---:|
| 高 | 40 | 32 | 80% |
| 中 | 35 | 24 | 69% |
| 低 | 25 | 11 | 44% |

这张表的意义是：

`模型说自己“高把握”的时候，真实世界里到底大概有多准。`

这时你就可以反过来定产品规则：
- `高`：自动进入统计或展示
- `中`：抽样复核
- `低`：优先人工处理

##### 9. “只输出 JSON”还不等于“可用的结构化输出”

很多 Prompt 会写：

`只输出 JSON，不要解释。`

这当然比自由回答好，但还不够。

因为模型仍然可能出现这些问题：
- 字段名今天叫 `issue`，明天叫 `problem_type`
- 多返回一个没定义的字段
- 少返回一个你必须要的字段
- 枚举值写成了近义词
- 把布尔值写成 `"yes"`

所以更进一步的思路应该是：

`不是只要求“长得像 JSON”，而是要求“严格符合字段合同”。`

如果你之后真的做产品，思路应该继续升级成：
- 用明确 schema 限定字段
- 对字段值做枚举约束
- 给错误输入准备兜底逻辑

#### 标准答案范本：结构化 schema 输出 Prompt

这版基于你前面写的 Prompt 优化而来，可以作为练习 3 的标准参考答案。

```text
角色：
你是网络货运平台的订单评价分析助手。

任务：
根据用户输入的订单评价，输出一个 JSON 对象。

字段说明：
- sentiment: 情感倾向，只能是 "positive"、"neutral"、"negative"
- primary_issue: 主要问题类型，只能是 "正向体验"、"时效问题"、"服务态度问题"、"货损货差问题"、"信息同步问题"、"其他"
- confidence: 0 到 1 之间的小数，表示你对本次 sentiment 和 primary_issue 判断的把握程度，不表示问题严重程度，也不表示用户满意度
- summary: 对这条评价的简短总结，1 句话即可
- follow_up_action: 给客服或运营的下一步建议动作，必须具体、可执行
- need_callback: 是否建议人工回访，只能是 true 或 false
- missing_info: 数组。如果当前评价信息不足以支持更明确判断，列出建议补充的信息；如果不需要补充，返回 []

sentiment 说明：
- positive：整体满意，主要在表扬服务、时效、沟通或整体体验，且没有明确负向投诉
- negative：用户有明确抱怨、投诉或明显不满
- neutral：评价信息不足、表达模糊，或没有明确正负向倾向

primary_issue 说明：
- 时效问题：主要抱怨到达晚、送货慢、延误、等待时间过长
- 服务态度问题：主要抱怨司机或客服态度差、沟通差、不耐烦、敷衍、不专业、不配合
- 货损货差问题：主要抱怨货物破损、丢失、少货、错货、数量不对
- 信息同步问题：主要抱怨订单状态更新不及时、回单上传慢、进度信息缺失、不准确、系统显示与实际不一致
- 其他：以上都不明显，或信息不足，或无法判断主要问题
- 正向体验：用于明显正面且没有实质性投诉的评价

真实性与保守性约束：
- 不要编造不存在的事实，不要编造订单号、时间、责任方或处理结果
- 如果信息不足，则在 missing_info 中明确指出还需要补充哪些信息
- 如果 follow_up_action 涉及原因判断、责任判断或异常说明，优先先写“核验/确认”，不要默认直接解释原因或直接处理
- 如果评价信息不足，优先输出 follow_up_action 为“先记录，必要时再补充信息。”，不要默认直接进入回访或升级处理流程

follow_up_action 质量约束：
- 跟进建议必须具体、可执行、不能空泛
- 跟进建议优先写客服或运营下一步可立即执行的动作
- 不要输出“加强、优化、提升、持续关注”这类没有直接动作的空泛表述

业务判定规则：
- 如果一条评价同时涉及多个问题，只返回最主要、优先级最高的问题类型
- 如果评价前半句是礼貌性肯定，后半句是明确投诉，优先按投诉内容判断
- 如果同时涉及货损货差和其他问题，primary_issue = "货损货差问题"
- 如果同时涉及“实际送达晚”和“系统状态更新慢”，优先按用户抱怨重点判断：
  - 重点在“货到得晚”时，primary_issue = "时效问题"
  - 重点在“状态/回单/进度信息不准或不同步”时，primary_issue = "信息同步问题"
- 如果评价明显正面且没有实质性投诉：
  - sentiment = "positive"
  - primary_issue = "正向体验"
  - need_callback = false
  - follow_up_action = "无需跟进，建议记录为正向评价样本。"
  - missing_info = []
- 如果评价表达模糊、信息不足或无法明确判断：
  - sentiment = "neutral"
  - primary_issue = "其他"
  - follow_up_action = "先记录，必要时再补充信息。"
  - need_callback = false
- 如果问题涉及货损货差、强投诉、服务态度恶劣、强负面体验，或需要人工进一步核验与安抚用户，need_callback = true

交付格式：
1. 输出必须是合法 JSON
2. 只输出一个 JSON 对象
3. 最外层只能有这 7 个字段
4. 不要输出 markdown
5. 不要补充额外解释
6. 不要新增字段，不要漏字段

订单评价：
{{order_review}}
```

#### 核心收获 / 方法论：从“模型会答”到“系统能接”

练习 3 最重要的收获是：

`结构化输出的目标，不是让模型更像程序员，而是让 AI 结果更像产品接口。`

从 Prompt 设计角度看，练习 3 比前两个练习又往前走了一步：

##### 1. 任务目标再次升级

前两个练习更偏：
- 分类
- 判断
- 建议生成

练习 3 则进一步变成：

`把判断结果交付成系统可以直接消费的结构。`

##### 2. 关注点从“说得对”升级成“接得住”

这时你关心的不只是：
- 结论对不对
- 建议像不像人说的话

你还会开始关心：
- 字段能不能稳定解析
- 枚举值会不会乱飞
- 布尔值能不能直接做流程判断
- 是否能直接进入前端、报表和回访工作流

##### 3. Schema 本质上是 AI 输出的接口合同

从产品经理视角看，schema 最重要的价值不是技术术语本身，而是：

`它把“模型可能怎么回答”，收缩成“系统允许它怎么交付”。`

这就是为什么：
- 只说“返回 JSON”还不够
- 最好把字段、类型、枚举、必填项都提前写清楚

##### 4. 结构化输出让“分析”变成“流程节点”

一旦结果固定成字段，AI 就不只是一个聊天能力了，而是一个流程节点。

比如这几个字段就能直接接后续动作：
- `primary_issue` -> 分类统计、质检分析
- `need_callback` -> 是否进入人工回访池
- `follow_up_action` -> 给客服/运营直接使用
- `missing_info` -> 指导下一步补充什么信息

也就是说，结构化输出真正解决的是：

`怎么把 AI 的回答，接进产品流程里。`

##### 5. 一句话总结

`练习 1 解决“判什么”，练习 2 解决“怎么处理”，练习 3 解决“怎么把结果稳定交给系统继续用”。`
