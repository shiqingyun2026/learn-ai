Roadmap

Day 2 到 Day 4，到底在学同一件什么事

这三天不是三套互相独立的 Prompt 课，而是在逐步搭一套更像产品系统的模型规范。

Day 2

守住知识边界

重点不是让模型更会说，而是让它只基于资料回答、资料不足时承认不足、不要把常识脑补成企业规则。

  • 先定义输出合同
  • 资料不足时明确兜底
  • 结构化输出让系统接得住
Day 3

选对知识路线

不是所有问题都上 RAG。要先判断规则少不少、资料大不大、答案是否要追溯、知识是不是需要长期沉淀。

  • Prompt 塞规则适合小而稳的任务
  • 长上下文适合一次性材料
  • RAG / Wiki 适合企业知识场景
Day 4

给模型接上行动能力

重点不是“能不能调工具”，而是“什么时候该查、先查什么、什么时候必须停下来或转人工”。

  • Workflow 管流程
  • Agent 管决策
  • Tool Calling 管执行
Method

Prompt 设计方法论

Prompt 不是一句问法，而是任务、规则、输出、边界和兜底动作的组合设计。真正稳定的 Prompt，往往来自方法，而不是灵感。

最佳设计流程

1
先定义任务明确要解决什么问题，输入是什么，输出给谁用。
2
先定义输出合同先定标签、字段、枚举、兜底值，再写 Prompt。
3
先定义评测标准没有评测，优化就会变成“凭感觉改文案”。
4
先跑测试，再补规则先找错误模式，再去补边界、优先级、格式和兜底。
5
只在必要时补 few-shotfew-shot 用来补最难样本，不是为了把 Prompt 变长。

写规则时的 5 个优先级

1. 输出闭集约束 先收住标签、字段、枚举值，防止模型自由发挥。
2. 防止乱编约束 资料不足、信息不足、证据不足时，怎么保守处理。
3. 建议质量约束 必须具体、可执行，不能空泛。
4. 业务判定规则 多问题怎么判主问题，冲突场景如何优先级处理。
5. 输出格式约束 是否必须 JSON，是否允许额外解释，系统如何接。
Evolution

从任务提示词，到工具调用提示词

这条进化路线最重要的，不是 Prompt 越写越长，而是它越来越像一套系统规范。

Step 1
任务提示词

先回答“这是什么任务”，例如分类、提取、摘要、建议。这个阶段重点是把目标和输出写清楚。

Step 2
结构化输出提示词

再回答“输出要长什么样”。从自由文本，收成标签、字段或 JSON，让系统能解析、统计和接流程。

Step 3
知识增强提示词

接着回答“知识从哪里来”。是直接把规则写进 Prompt，还是用长上下文、RAG 或 Wiki 去补知识。

Step 4
工具调用提示词

最后回答“模型能行动到哪一步”。什么时候查工具、查哪些、够不够、哪些动作不能自动执行。

Types

你最近其实在写这 6 类 Prompt

分类类 Prompt

闭集

把文本稳定映射到固定标签集合里。

  • 先收住标签
  • 定义边界和优先级
  • 信息不足要有兜底

建议类 Prompt

动作

不只是看懂内容，还要输出下一步可执行动作。

  • 建议必须具体
  • 高风险先核验或升级
  • 不要输出空泛话术

结构化输出 Prompt

Schema

让输出能被系统接住，而不是只变成一段漂亮文字。

  • 先定义字段
  • 再定义 schema
  • 固定合法格式

检索型 Prompt

RAG

只基于检索资料作答，资料不足就承认不足，资料冲突就承认冲突。

  • 只在证据内回答
  • 不要把常识补成规则
  • 尽量绑定材料来源

Wiki / 汇总型 Prompt

归纳

基于整理后的知识层做总结和解释，更强调归纳质量和口径一致性。

  • 区分原文与总结
  • 指出资料差异
  • 精确条款回查原文

Tool Calling Prompt

行动

让模型知道什么时候查工具、什么时候停、什么时候转人工。

  • 调用前置条件
  • 最少必要工具原则
  • 动作类工具边界
Templates

可直接练习的 Prompt 模板

下面不是完整成品，而是最适合入手的一版最小模板。先用它们起稿，再按 badcase 补规则。

模板 1 · 分类类 Prompt 适合主题分类、意图识别、工单归类等固定标签任务。
01
先收住标签闭集，再补边界和优先级，最后把输出收成固定值。
角色：
你是[业务场景]的[分类助手名称]。

任务：
根据用户输入的[文本类型]，判断其最主要的[分类对象]，并从以下标签中选择一个返回。

标签定义：
- [标签1]：[定义]
- [标签2]：[定义]
- [标签3]：[定义]
- 其他：[兜底定义]

判定规则：
1. 只能选择一个最主要标签
2. 如果一条内容同时涉及多个问题，只返回用户最核心抱怨对应的标签
3. 如果信息不足，返回“其他”

输出要求：
1. 你的回答必须且只能是标签列表中的一个值
2. 不要解释原因
3. 不要输出额外文字

输入内容：
{{input_text}}
模板 2 · 建议类 Prompt 适合客服建议、运营跟进、异常处理建议等动作型任务。
02
建议类 Prompt 的核心不是像人说话，而是输出能被执行的下一步动作。
角色：
你是[业务场景]的[分析助手名称]。

任务：
根据用户输入的[文本类型]，识别主要问题，并给[使用对象]输出具体、可执行的跟进建议。

一、输出闭集约束
1. 问题类型必须且只能从以下标签中选择：[标签A]、[标签B]、[标签C]、其他

二、真实性与保守性约束
2. 不要编造不存在的事实
3. 如果信息不足，明确指出还需要补充哪些信息
4. 如果建议涉及原因判断、责任判断或异常说明，优先先写“核验/确认”

三、建议质量约束
5. 跟进建议必须具体、可执行、不能空泛
6. 跟进建议优先写[使用对象]下一步可立即执行的动作

四、业务判定规则
7. 如果是高风险问题，优先给出回访、核验或升级处理动作

五、输出格式约束
8. 请按固定格式输出，不要补充额外解释
模板 3 · 结构化输出 Prompt 适合需要前端展示、流程判断和系统接入的 JSON 输出任务。
03
先定义字段和 schema，再写 Prompt。结构化输出让模型从“会答”变成“能接流程”。
角色：
你是[业务场景]的[分析助手名称]。

任务：
根据用户输入的[文本类型]，输出一个 JSON 对象，供前端展示、运营跟进和后续流程判断使用。

字段说明：
- [field_1]：[类型 / 含义 / 枚举]
- [field_2]：[类型 / 含义 / 枚举]
- [field_3]：[类型 / 含义 / 枚举]

真实性与保守性约束：
- 不要编造不存在的事实
- 如果信息不足，在缺失字段中明确指出需要补充什么

交付格式：
1. 输出必须是合法 JSON
2. 只输出一个 JSON 对象
3. 不要新增字段，不要漏字段
4. 不要输出 markdown 或额外解释
模板 4 · 检索型 Prompt 适合 SOP、规则、FAQ、资料问答等基于检索上下文的回答任务。
04
检索型 Prompt 的第一职责，不是更会答，而是只在资料支持的范围内答。
请仅基于给定资料回答用户问题，不要使用资料外的信息，不要补充或假设企业规则。

回答要求：
1. 优先依据资料中的明确内容直接回答问题
2. 如果资料不足以支持明确结论，必须明确回答“当前资料不足，无法确认”
3. 如果资料之间存在冲突，必须明确说明“资料内容存在冲突”
4. 不要编造规则、时间、责任方、处理结论或资料中未明确说明的限制条件
5. 回答尽量简洁，先给结论，再说明依据

资料：
{{retrieved_context}}

问题：
{{user_query}}
模板 5 · Tool Calling Prompt 适合内部客服/运营助手、异常处理助手、任务执行型 Agent。
05
重点不是让模型多调用工具，而是写清：什么时候查、先查什么、什么时候停、什么时候转人工。
角色：
你是[业务场景]内部使用的[助手名称]。

任务：
根据输入内容，结合必要的上下文和工具结果，输出结构化判断与下一步建议。

可调用工具：
- [tool_1]
- [tool_2]
- [tool_3]

工具使用规则：
1. 先基于输入直接判断；只有在需要更准确判断或生成更可靠建议时，才调用工具
2. 优先调用查询类工具，先补全上下文，再决定是否继续
3. 只调用最少必要工具，不要重复调用，不要无意义扩查
4. 动作类工具必须更谨慎使用，只能在高风险、信息冲突、需要人工核验等场景下调用

高风险边界：
1. 不要直接给出赔付结论、责任归属或对外承诺
2. 信息不足时优先返回“信息不足”或建议人工核验
3. 高风险场景必须转人工或创建人工复核工单

交付格式：
1. 输出必须是合法 JSON
2. 只输出一个 JSON 对象
3. 最外层字段固定
4. 不要补充额外解释
Boundaries

工具调用最容易出问题的地方

Day 4 最关键的，不是学会几个新术语，而是知道为什么 Prompt 里必须把工具边界和停手条件写清楚。

输入
“这单体验很差”

这是一个典型的低信息输入。用户表达了不满，但没有明确说清是时效、态度、货损还是信息同步问题。

产品问题：系统应该继续乱查，还是先停下来？
无边界系统
模型会倾向继续行动

如果 Prompt 没写最少必要工具原则，它可能会把订单信息、状态、规则、SOP 都查一遍，试图“把事情做完整”。

结果：调用过多、成本增加、还可能因为证据不足而判断跑偏。
产品级系统
先判断是否足够，再决定查不查

更合理的逻辑是：先判断信息是否足够；不足就返回信息不足，或者直接进入人工复核，而不是无意义扩查。

结果：更可控、更省成本，也更符合高风险业务的保守策略。

没有“最少必要工具原则”时

模型的生成机制，是根据上下文去预测下一个最合理的输出。训练里我们也经常鼓励模型继续推进任务，所以它天然更容易继续查、继续想、继续做。

  • 能查一个工具时查三个
  • 明明信息不足却继续推进
  • 为了“更完整”而无意义扩查
  • 更容易出现乱用工具和多步失控

写清工具边界后

产品上更合理的办法，是让模型先判断信息是否足够。若不足，就停下来、返回信息不足，或者直接进入人工复核。

  • 先判断是否真的需要查工具
  • 优先查询类工具，动作类工具后置
  • 足够就停，不要重复调用
  • 高风险场景直接转人工

例子：如果用户评价只有一句 “这单体验很差”，没有边界的模型可能会把订单基础信息、订单状态、正常流转规则、异常处理 SOP 都查一遍；但更合理的产品逻辑是：先判断信息是否足够，不足就返回 信息不足，或直接进入人工判断。

Concept 1

tool schema

给模型看的“工具说明书 + 参数格式说明”。

Concept 2

strict mode

让模型更严格地按 schema 传参数，少乱补字段和类型。

Concept 3

call_id

工具调用链路里的“对账编号”，把调用请求和调用结果对应起来。

Concept 4

response 结构

response 不只是最后一句回复，还包含工具调用过程和中间结果。

Final Form

Prompt 的最终形态，不是一句话，而是一套 Stack

真正可落地的 AI Prompt，最后都会越来越像一套系统规范。不是文案更花，而是任务、输出、知识、工具和人工停手条件都被写进去了。

角色与任务定义

先让模型知道自己是谁、要解决什么问题、这次任务服务于谁。

输出 schema

先定义字段、枚举、必填项和兜底值，让模型输出能被系统接住。

知识层约束

回答能不能用资料外信息，知识从 Prompt、长上下文、RAG 还是 Wiki 来。

工具调用规则

哪些工具可用，先查什么，调用前需要哪些参数和业务前提。

高风险边界

哪些结论不能自动下，哪些动作不能自动做，哪些场景必须保守处理。

人工兜底规则

信息不足、资料冲突、低置信度和高风险场景下，模型应该停在哪里。

Checklist

写 Prompt 前，先问自己的 8 个问题

任务面

  • 这个 Prompt 到底要解决什么任务？
  • 输出是给人看，还是给系统接？
  • 哪些字段必须先定义，哪些值必须闭集？
  • 我要不要先跑最小 badcase 集？

知识与工具面

  • 回答能不能使用资料外知识？
  • 这个任务更适合 Prompt、长上下文、RAG 还是 Wiki？
  • 什么时候该查工具，什么时候不该查？
  • 信息不足和高风险场景下，模型应该停在哪里？
